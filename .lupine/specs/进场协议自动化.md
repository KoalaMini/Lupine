# 进场协议自动化

> v0.8 · Lupine 新 session 时自动执行进场协议，无需用户提示

---

## 为什么需要这个

v0.7 在 AGENT.md 中定义了三层进场协议（架构层/需求层/执行层），但它是**文件层面的约定**——Agent 没有主动执行的指令。每次新 session 时：

- Agent 不知道"我应该先读 AGENT.md"
- 用户需要手动提示或 Agent 靠经验猜到该做什么
- 没有标准化的背景确认环节，用户不知道 AI 是否了解项目

v0.8 的目标是：**新 session 开始时，Lupine 自动执行进场协议，并以自然语言告知用户已就绪。**

---

## 范围

### 列入范围

1. **Lupine prompt 增加"进场协议"指令** — 新 session 时自动执行 AGENT.md 中定义的三层协议
2. **Planner / Executor prompt 增加上下文继承说明** — 子 Agent 不再自主读取项目文档
3. **`lupine init` 源码探查** — 初始化时自动生成 ARCHITECTURE.md，支持 5 种语言，合并探查多 repo
4. **预览确认机制** — init 时展示探查结果，用户确认后写入

### 不列入范围

- `lupine update` 探查 ARCHITECTURE.md — 现有版本是手工维护的，自动覆盖体验不佳
- 评估器 prompt 不改 — 评估器不需要项目上下文
- 子 Agent 的独立进场协议 — 由 Lupine 口头传递上下文
- 进场指示灯的输出格式强制 — 由 Lupine 用自然语言表达
- 架构约束自动化 linter — 独立方向
- 语言探查插件机制 — v0.8 硬编码支持 5 种语言

---

## 需求规格

### REQ-1: Lupine 进场协议指令

**位置**：Lupine prompt（`lupine.prompt` / `lupine.md`）末尾，新增"进场协议"段落。

**行为**：
- 新 session 开始时，在执行"通情达理"法之前，先执行 AGENT.md 的三层恢复流程
- 完成后以自然语言告知用户已了解背景，例如：
  > "我已了解项目架构、技术栈、当前功能状态……我们这次要讨论什么？"
- 同 session 连续对话不重复执行

**调度影响**：调度子 Agent 时，确保子 Agent 知道其所需的上下文信息（由 Lupine 口头传递，非通过文件读取）。

### REQ-2: Planner / Executor prompt 微调

**位置**：各 prompt 末尾新增一句话。

**内容**：
```
上下文由调度器 Lupine 传递。不需要自行读取项目文档。
```

**行为变化**：
- Planner 不再自主读取 specs/ 外的项目文件
- Executor 不再自主读取 rules/ 外的项目文件
- 所有上下文来源：Lupine 调度时的指令 + 传入的 spec/plan 引用

### REQ-3: 源码探查引擎（probe.js）

**新增文件**：`packages/lupine/src/probe.js`

**输入**：一个或多个 repo 路径

**输出**：结构化探查结果，包含：
- `lang` — 检测到的语言（node / python / go / rust / java / unknown）
- `name` — 项目名（从标志文件提取，如 package.json#name）
- `description` — 项目描述（从标志文件提取）
- `deps` — 关键依赖数组
- `scripts` — 可用脚本（从标志文件提取）
- `dirTree` — 目录树（深度 2-3 层）
- `ciDetected` — 是否检测到 CI 配置

**语言检测**：

| 语言 | 检测文件 |
|------|---------|
| Node.js | `package.json` |
| Python | `pyproject.toml` / `setup.py` / `requirements.txt` |
| Go | `go.mod` |
| Rust | `Cargo.toml` |
| Java | `pom.xml` / `build.gradle` |
| 未知 | 无匹配 |

**降级策略**：未匹配任何语言时，仅输出目录树，技术栈标记为"未识别"。

**目录树深度**：2 层（根目录 + 一级子目录），最多展到 3 层（当子目录夹带 README.md 等关键文件时）。

### REQ-4: init 中的探查集成

探查仅在 `lupine init` 时触发，`lupine update` 不涉及 ARCHITECTURE.md 的改动（现有版本是手工精心维护的，自动覆盖体验不佳）。

**`lupine init`**：
- 在现有模板复制流程完成后，对 repositories 中所有 repo 逐一探查
- 生成 ARCHITECTURE.md，展示探查结果给用户
- 用户确认（Y/n）后写入 `.lupine/ARCHITECTURE.md`

**ARCHITECTURE.md 结构**（保持 v0.7 三段式）：

```markdown
# 项目架构

## 产品架构
[多 repo 合并描述]

## 技术架构
[多 repo 分组列出，每个 repo 独立段落]

## 目录结构
[多 repo 分别展示]

## 基础设施
[多 repo 命令合并列表]
```

**预览展示方式**：终端 diff 风格（新增行绿色、删除行红色、无变化不展示），附最终完整内容供确认。

---

## 验收条件

1. **新 AI 会话（新 session）** → Lupine 自动执行三层协议，输出自然语言"我已了解..."，用户无需提示
2. **新项目跑 `lupine init`** → 自动生成含仓库信息的 ARCHITECTURE.md，确认后写入
3. **Node / Python / Go / Rust / Java 项目** → 均能正确识别技术栈
4. **无标志文件的项目** → 优雅降级，输出目录树，技术栈标"未识别"
5. **Planner / Executor 被调度时** → 不自主读取项目文档，按 Lupine 传递的上下文工作
