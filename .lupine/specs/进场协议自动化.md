# 进场协议自动化

> v0.8 → v0.9 · 从前置仪式演进为"反射"子步骤，上下文唤醒融入通情达理

---

## 为什么需要这个

进场协议经历了两个阶段的认知迭代：

**v0.7** 在 AGENT.md 中定义了三层协议（架构层/需求层/执行层），但仅停留在文件约定，Agent 没有主动执行指令。

**v0.8** 在 prompt 末尾增加"进场协议"段落，让 Agent 在收到用户第一条消息后前置执行 8 步 I/O。但仍有一个核心矛盾：**设计意图是"新 session 自动执行"，实际必须等用户先发消息才能触发**。仪式与对话是串行的两个阶段，用户感知到"先加载，后沟通"的割裂感。

**v0.9** 发现问题本质：上下文恢复不应该是通情达理之前的独立阶段，而是**反射步骤的细化**。用户的第一条消息触发对话（陈述），Lupine 在回应时自动唤醒上下文并融合到对需求的理解中（反射）。对话流从：

```
用户消息 → 仪式(8步) → 陈述→反射→发散...
```

变为：

```
用户消息 → 陈述 → 反射(上下文唤醒 + 需求反射) → 发散...
```

---

## 范围

### 列入范围

1. **反射细化** — 上下文唤醒从独立前置步骤变为"反射"的子步骤，同 session 仅首次触发
2. **Planner / Executor prompt 增加上下文继承说明** — 子 Agent 不再自主读取项目文档
3. **`lupine init` 源码探查** — 初始化时自动生成 ARCHITECTURE.md，支持 5 种语言，合并探查多 repo
4. **预览确认机制** — init 时展示探查结果，用户确认后写入

### 不列入范围

- `lupine update` 探查 ARCHITECTURE.md — 现有版本是手工维护的，自动覆盖体验不佳
- 评估器 prompt 不改 — 评估器不需要项目上下文
- 子 Agent 的独立入场流程 — 由 Lupine 口头传递上下文
- 上下文唤醒输出指示灯 — 产出融合到反射中，不独立输出
- 架构约束自动化 linter — 独立方向
- 语言探查插件机制 — v0.8 硬编码支持 5 种语言

## 需求规格

### REQ-1: 反射细化 — 上下文唤醒

**位置**：Lupine prompt（`lupine.prompt` / `lupine.md`）的"通情达理"法第 2 步"反射"中，增加子步骤 a。

**行为**：
- 对话首次响应时，在反射阶段自动执行上下文唤醒：
  - 读 AGENT.md → 规范与边界
  - 读 ARCHITECTURE.md → 架构与技术栈
  - 读 FEATURES.json → 功能状态
  - `git log --oneline -20` → 近期变更
  - 读 plans/ → 进行中的计划
- 同 session 后续对话不再重复唤醒
- 唤醒产出不独立输出，而是融合到对用户需求的理解中一并表达

**输出示例**：
```
基于 Lupine 项目当前状态（F4 构建交付进行中、F7 进场协议已完成），
你提的这个需求属于新功能方向。我的理解是……
```

**调度影响**：调度子 Agent 时，确保子 Agent 知道其所需的上下文信息（由 Lupine 口头传递，非通过文件读取）。

### REQ-2: Planner / Executor prompt 微调

**位置**：各 prompt 末尾新增一句话。

**内容**：
```
上下文由调度器 Lupine 传递。不需要自行读取项目文档。
```

**行为变化**：
- Planner 不再自主读取 specs/ 外的项目文件
- Executor 不再自主读取 rules/ 外的项目文件
- 所有上下文来源：Lupine 调度时的指令 + 传入的 spec/plan 引用

### REQ-3: 源码探查引擎（probe.js）

**新增文件**：`packages/lupine/src/probe.js`

**输入**：一个或多个 repo 路径

**输出**：结构化探查结果，包含：
- `lang` — 检测到的语言（node / python / go / rust / java / unknown）
- `name` — 项目名（从标志文件提取，如 package.json#name）
- `description` — 项目描述（从标志文件提取）
- `deps` — 关键依赖数组
- `scripts` — 可用脚本（从标志文件提取）
- `dirTree` — 目录树（深度 2-3 层）
- `ciDetected` — 是否检测到 CI 配置

**语言检测**：

| 语言 | 检测文件 |
|------|---------|
| Node.js | `package.json` |
| Python | `pyproject.toml` / `setup.py` / `requirements.txt` |
| Go | `go.mod` |
| Rust | `Cargo.toml` |
| Java | `pom.xml` / `build.gradle` |
| 未知 | 无匹配 |

**降级策略**：未匹配任何语言时，仅输出目录树，技术栈标记为"未识别"。

**目录树深度**：2 层（根目录 + 一级子目录），最多展到 3 层（当子目录夹带 README.md 等关键文件时）。

### REQ-4: init 中的探查集成

探查仅在 `lupine init` 时触发，`lupine update` 不涉及 ARCHITECTURE.md 的改动（现有版本是手工精心维护的，自动覆盖体验不佳）。

**`lupine init`**：
- 在现有模板复制流程完成后，对 repositories 中所有 repo 逐一探查
- 生成 ARCHITECTURE.md，展示探查结果给用户
- 用户确认（Y/n）后写入 `.lupine/ARCHITECTURE.md`

**ARCHITECTURE.md 结构**（保持 v0.7 三段式）：

```markdown
# 项目架构

## 产品架构
[多 repo 合并描述]

## 技术架构
[多 repo 分组列出，每个 repo 独立段落]

## 目录结构
[多 repo 分别展示]

## 基础设施
[多 repo 命令合并列表]
```

**预览展示方式**：终端 diff 风格（新增行绿色、删除行红色、无变化不展示），附最终完整内容供确认。

## 验收条件

1. **新 AI 会话（新 session）** → 用户发送第一条消息后，Lupine 在反射阶段自动唤醒上下文并融合到对需求的理解中，无需独立"我已了解"仪式
2. **同 session 连续对话** → 第二次起不再重复上下文唤醒
3. **新项目跑 `lupine init`** → 自动生成含仓库信息的 ARCHITECTURE.md，确认后写入
4. **Node / Python / Go / Rust / Java 项目** → 均能正确识别技术栈
5. **无标志文件的项目** → 优雅降级，输出目录树，技术栈标"未识别"
6. **Planner / Executor 被调度时** → 不自主读取项目文档，按 Lupine 传递的上下文工作
