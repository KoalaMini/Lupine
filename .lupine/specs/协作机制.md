# 协作机制

> Lupine 自主决策调度 + specs∥plans 并行文档体系

---

## 1. 概述

Lupine 的协作机制核心原则是：**Lupine 自主决策，角色边界硬约束**。没有预设的固定流水线，Lupine 根据任务类型、复杂度、风险自主决定调度策略，但所有子 Agent 必须遵守角色边界硬规则。

---

## 2. 调度模型

### 2.1 核心原则

- **Lupine 是唯一调度者**：用户只与 Lupine 对话，Lupine 决定何时调用哪个子 Agent
- **不预设路径**：没有固定的"先规划再执行"流水线，简单任务可以跳过规划
- **约束优先**：无论怎么调度，角色边界硬规则始终生效

### 2.2 典型调度场景

| 场景 | Lupine 的决策 | 涉及角色 |
|------|--------------|---------|
| **新项目第一个功能** | 完整流水线：探讨 → 探索 → 规划 → 执行 → 评估 | 全部 |
| **已有项目加功能** | 完整流水线 | 全部 |
| **修bug** | plan  → review plan → code → review code | 执行器 + 评估器 |
| **改文案/文档** | 直接执行后审查 | 执行器 + 评估器 |
| **技术调研** | 仅调用探索器 | 探索器 |

### 2.3 调度流程示例（完整流水线）

```
用户提需求
    ↓
Lupine 澄清需求、探讨方案、写 specs/
    ↓
Lupine 调用探索器（可选）→ 代码库调研、技术选型
    ↓
Lupine 调用规划器 → 写 plans/
    ↓
评估器审查 plan → 通过/打回
    ↓
Lupine 调用执行器 → 按 plan 写 code + tests
    ↓
执行器自测 → 提交
    ↓
评估器审查 code → 通过/打回
    ↓
Lupine 审查 review → 决定是否合并
    ↓
用户验收
```

**打回机制**：任何阶段被评估器打回，都会回到对应的上一步重做。

## 3. 并行文档体系

### 3.1 specs∥plans 并行

**核心设计**：specs（需求规格）和 plans（执行计划）是**并行**的，不是串行的。

- **specs/**：回答 WHAT + WHY，由 Lupine 维护
- **plans/**：回答 HOW，由规划器维护
- 一个 spec 可以对应多个 plan（不同实现方案）
- 一个 plan 只能对应一个 spec（不可跨 spec）

### 3.2 文档关系

```
PRODUCT.md（产品总纲）
    ├── specs/需求A.md（WHAT+WHY）
    │       ├── plans/方案1.md（HOW）
    │       └── plans/方案2.md（HOW，备选）
    ├── specs/需求B.md
    │       └── plans/方案1.md
    └── reviews/
            ├── 方案1-review.md
            └── 方案2-review.md
```

### 3.3 文档生命周期

| 阶段 | 动作 | 负责人 |
|------|------|--------|
| 需求确认 | 写 spec | Lupine |
| 方案设计 | 写 plan | 规划器 |
| plan 审查 | review plan | 评估器 |
| 代码实现 | 写 code | 执行器 |
| code 审查 | review code | 评估器 |
