# 上下文体系

> 产出物模型、规范文件体系、Agent 定义体系、工作区配置、功能清单

---

## 1. 概述

上下文体系定义 Lupine 的所有**文档产出物**和**配置文件**，确保项目知识可持续、可追溯、可恢复。所有文档有明确的所有者、读者和更新规则。

---

## 2. 产出物模型

### 2.1 核心产出物

| 文档 | 读者 | 所有者 | 更新频率 | 回答的问题 |
|------|------|--------|---------|-----------|
| **PRODUCT.md** | AI + 人 | Lupine | 产品定位变化时 | 产品是什么？为谁而做？ |
| **proposal/** | 人（决策者） | Lupine | 技术选型/架构决策时 | 应该选哪个方案？ |
| **specs/** | 人（PM/开发者） | Lupine | 需求变更时 | WHAT + WHY |
| **plans/** | AI（执行器为主） | 规划器 | plan 不可行时 | HOW |
| **reviews/** | 人 + AI | 评估器 | 每次审查后 | 质量如何？ |
| **FEATURES.json** | AI（Agent） | Lupine | 功能变更时 | 项目有哪些功能？状态如何？ |

### 2.2 文档定义

**PRODUCT.md（产品总纲）**
- 位置：项目根目录
- 内容：产品定位、核心理念、目标用户、成功标准、范围边界
- 规则：只有 Lupine 能更新，其他角色只读

**specs/（需求规格）**
- 位置：`specs/` 目录
- 内容：按大类组织的 spec 文档（角色模型、协作机制、上下文体系、构建交付）
- 规则：Lupine 负责撰写和维护，其他角色只读

**proposal/（提案）**
- 位置：`proposal/` 目录
- 内容：技术选型对比、架构方案评审、中长期规划等调研型产出
- 生命周期：决策后可能归档，与 specs/ 独立演进
- 规则：仅 Lupine 可写，其他角色只读

**plans/（执行计划）**
- 位置：`plans/` 目录
- 内容：任务拆分、依赖关系、验收标准
- 规则：规划器撰写，Lupine 和执行器只读

**reviews/（审查记录）**
- 位置：`reviews/` 目录
- 内容：评估器的审查结果（通过/不通过 + 具体原因）
- 规则：评估器撰写，所有角色只读

**FEATURES.json（功能清单）**
- 位置：项目根目录
- 内容：功能全景 + 状态 + 历史变更摘要
- 用途：Agent 多会话上下文恢复、回归测试参考
- 规则：功能变更时更新

## 3. 规范文件体系

### 3.1 文件清单

所有规范文件聚合在 `rules/` 目录：

| 文件 | 定位 | 读者 | 核心内容 |
|------|------|------|---------|
| `templates/agents/` | Agent prompt 模板 | 渲染为各 AI 工具的 Agent 配置 | 每个 Agent 的 prompt、MCP/Skill、约束 |
| `rules/coding.md` | 编码规范 | 执行器 | 技术栈、命名规范、测试要求 |
| `rules/git.md` | Git 规范 | 所有 Agent | 分支策略、Commit 规范、PR 流程 |

### 3.2 约束注入

约束直接内联在各 Agent 的 prompt 中，无需外部 YAML 文件。

## 4. Agent 定义体系

### 4.1 源模板

Agent prompt 模板存放在 `packages/lupine/templates/agents/`：

```
packages/lupine/templates/agents/
├── _agents.json            ← 多平台元数据（opencode/Claude 配置）
├── lupine.prompt           ← Lupine 提示词模板
├── lupine-planner.prompt   ← 规划器提示词模板
├── lupine-executor.prompt  ← 执行器提示词模板
└── lupine-evaluator.prompt ← 评估器提示词模板
```

### 4.2 渲染目标

根据使用的 AI 编码工具，模板渲染到不同位置：

| 工具 | 目标目录 | 格式 |
|------|---------|------|
| opencode | `.opencode/agents/*.md` | Markdown + frontmatter |
| Claude Code | `.claude/agents/*.md` | Markdown + frontmatter |
| 其他 | 按工具规范适配 | 工具指定的格式 |

### 4.3 文件结构

每个 Agent 定义文件包含：
- **frontmatter**：工具特定配置（mode、permissions、tools）
- **prompt 正文**：角色定义、工作流、约束（直接内联）

## 5. 工作区配置

![三层隔离架构](上下文体系.assets/三层隔离架构.svg)

### 5.1 工作区结构

`.lupine/` 是 Lupine 的**AI 工作区目录**，存放配置、产出物和规则。它与用户代码仓库通过 `.lupineconfig.json` 中的 `repositories` 统一配置管理。

```
.lupine/
├── .lupineconfig.json       ← 配置（workspace、repositories、版本等）
├── CLAUDE.md                ← AI 索引入口
├── PRODUCT.md               ← 产品总纲
├── README.md                ← 项目介绍
├── rules/                   ← 规范文件
│   ├── coding.md
│   ├── evals.md
│   └── git.md
├── specs/                   ← 需求规格
├── proposal/                ← 提案（技术选型、架构方案）
├── plans/                   ← 执行计划
└── reviews/                 ← 审查记录
```

### 5.2 配置规范（.lupineconfig.json）

#### 配置结构

```json
{
  "version": "0.6.0",
  "projectName": "MyProject",
  "platform": "opencode",
  "workspace": {
    "root": ".",
    "repository": "lupine-workspace"
  },
  "repositories": [
    {
      "name": "lupine-workspace",
      "path": ".lupine",
      "independentGit": false,
      "type": "workspace"
    },
    {
      "name": "frontend",
      "path": "frontend",
      "type": "frontend",
      "independentGit": true
    }
  ],
  "skills": { ... }
}
```

#### 字段说明

**workspace（必填）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `workspace.root` | string | 工作区根目录。所有路径的基准点，AI 执行命令前 `cd` 到该目录。通常为 `"."` |
| `workspace.repository` | string | 指向 `repositories` 中的 `name`，标识哪个 repository 是 AI 工作区 |

**repositories（必填，不能为空数组）**

必须至少包含一个 repository（即 AI 工作区自身）。所有仓库（包括 `.lupine`）统一在 `repositories` 中配置。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `repositories[].name` | string | 是 | 仓库标识名，唯一 |
| `repositories[].path` | string | 是 | 相对于 `workspace.root` 的路径 |
| `repositories[].independentGit` | boolean | 是 | 是否为独立 Git 仓库 |
| `repositories[].type` | string | 否 | 仓库类型：`workspace` / `frontend` / `backend` / `lib` / `app` / `shared` / `infra` |

**type 取值说明**

| 取值 | 含义 | AI 语义提示 |
|------|------|------------|
| `workspace` | AI 工作区 | 存放 AI 产出物（specs/plans/reviews），非用户代码 |
| `frontend` | 前端应用 | 使用前端技术栈，关注 UI/组件/路由 |
| `backend` | 后端服务 | 使用后端技术栈，关注 API/数据库/业务逻辑 |
| `lib` | 库/框架 | 关注 API 设计、兼容性、文档 |
| `app` | 全栈应用 | 前后端在同一个仓库 |
| `shared` | 共享模块 | 被其他仓库依赖的公共代码 |
| `infra` | 基础设施 | 部署/配置/脚本 |

#### 路径规则

1. **所有路径字段**（`workspace.root`、`repositories[].path`）均为**相对路径**
2. **AI 执行命令前**，先 `cd` 到 `workspace.root` 指定的目录
3. **读取/写入文件时**，路径格式为 `{repositories[].path}/子路径`

#### Git 操作规则

AI 根据文件路径判断所属 repository，再决定 Git 执行目录：

1. 找到文件路径匹配的 repository（最长前缀匹配）
2. 若 `independentGit: true`：在 `{workspace.root}/{repositories[].path}` 执行 git
3. 若 `independentGit: false`：在 `{workspace.root}` 执行 git

**示例**：
- 修改 `.lupine/specs/xxx.md` → 匹配 `lupine-workspace` repository
  - 若 `independentGit: false` → 在 `workspace.root` 执行 `git add .lupine/specs/xxx.md`
  - 若 `independentGit: true` → 先 `cd .lupine/`，再执行 `git add specs/xxx.md`
- 修改 `frontend/src/App.jsx` → 匹配 `frontend` repository
  - 若 `independentGit: true` → 先 `cd frontend/`，再执行 `git add src/App.jsx`

#### 典型场景配置

**场景 1：.lupine 嵌入项目，非独立 Git（最常见）**

```json
{
  "workspace": {
    "root": ".",
    "repository": "lupine-workspace"
  },
  "repositories": [
    {
      "name": "lupine-workspace",
      "path": ".lupine",
      "independentGit": false,
      "type": "workspace"
    },
    {
      "name": "my-app",
      "path": ".",
      "independentGit": false,
      "type": "app"
    }
  ]
}
```

- `.lupine/` 和项目代码在同一个 Git 仓库中
- 所有 Git 操作在项目根目录执行

**场景 2：.lupine 是 Git Submodule**

```json
{
  "workspace": {
    "root": ".",
    "repository": "lupine-workspace"
  },
  "repositories": [
    {
      "name": "lupine-workspace",
      "path": ".lupine",
      "independentGit": true,
      "type": "workspace"
    },
    {
      "name": "my-app",
      "path": ".",
      "independentGit": false,
      "type": "app"
    }
  ]
}
```

- `.lupine/` 是独立 Git 仓库（submodule）
- `.lupine/` 内文件的 Git 操作在 `.lupine/` 目录执行
- 项目代码的 Git 操作在项目根目录执行

**场景 3：多仓库模式**

```json
{
  "workspace": {
    "root": ".",
    "repository": "lupine-workspace"
  },
  "repositories": [
    {
      "name": "lupine-workspace",
      "path": ".lupine",
      "independentGit": true,
      "type": "workspace"
    },
    {
      "name": "frontend",
      "path": "frontend",
      "type": "frontend",
      "independentGit": true
    },
    {
      "name": "backend",
      "path": "backend",
      "type": "backend",
      "independentGit": true
    },
    {
      "name": "algorithm",
      "path": "algorithm",
      "type": "lib",
      "independentGit": true
    }
  ]
}
```

- `.lupine/` 和所有代码仓库都是独立 Git 仓库
- 一个需求可能同时修改多个仓库，AI 根据文件路径在对应仓库分别执行 git

### 5.3 初始化流程

```bash
npx lupine init <project-name>
```

1. 创建 `.lupine/` 目录
2. 复制模板文件（PRODUCT.md、rules/、specs/ 等）
3. 生成 `.lupineconfig.json`（含 `workspace` 和 `repositories` 配置）
4. 安装推荐的 MCP/Skill 依赖

### 5.4 更新流程

```bash
npx lupine update
```

1. 对比本地与远程模板版本
2. 显示变更列表（新增/修改/删除）
3. 用户确认后合并变更
4. 更新 `.lupineconfig.json` 中的版本号

## 6. 功能清单（FEATURES.json）

### 6.1 定位

FEATURES.json 是项目的**功能导航图**，不是详细设计文档。它回答两个问题：
1. 项目有哪些功能？当前状态如何？
2. 功能的历史演进是什么？

### 6.2 核心用途

- **Agent 多会话上下文恢复**：新 session 读取 FEATURES.json，快速了解项目全貌
- **回归测试参考**：验证功能完整性，防止遗漏
- **版本规划**：基于当前功能状态规划下一版本

### 6.3 数据结构

```json
{
  "manifest": { "version": "...", "lastUpdated": "...", "totalFeatures": N },
  "features": [
    {
      "id": "F1",
      "name": "角色模型",
      "status": "completed",
      "description": "...",
      "currentSpec": "specs/角色模型.md",
      "assetsRef": "specs/角色模型.assets/",
      "history": [
        { "version": "0.1", "change": "...", "status": "superseded" },
        { "version": "0.2", "change": "...", "status": "completed" }
      ]
    }
  ]
}
```

### 6.4 状态枚举

| 状态值 | 含义 | 适用范围 |
|--------|------|---------|
| `completed` | 已完成 | 当前功能已交付 |
| `in-progress` | 进行中 | 当前功能正在开发 |
| `planned` | 计划中 | 已规划但尚未开始 |
| `superseded` | 被替代 | 仅用于 history，表示该版本的功能已被后续版本取代 |

**规则**：
- `feature.status` 只能是 `completed` / `in-progress` / `planned`
- `history[].status` 可以是 `completed` / `superseded`
- `superseded` 仅用于历史记录，不可作为当前功能状态

### 6.5 维护规则

- **谁更新**：Lupine（功能变更时）
- **何时更新**：新增功能、功能状态变化、spec 重构时
- **维护工具**：由 `requirements-management` skill 提供具体流程
- **触发机制**：Lupine 在决策框架中判断变更类型后，加载 skill 执行
- **一致性检查**：FEATURES.json 的 `currentSpec` 必须与实际文件路径 1:1 对应
