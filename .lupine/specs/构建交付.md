# 构建交付

> npm 脚手架 CLI、模板同步与版本管理、双渠道发布

---

## 1. 概述

构建交付模块负责 Lupine 框架的分发和安装。用户通过 npm 安装 CLI 工具，一键初始化项目工作区，并通过模板同步保持与上游框架的更新。

---

## 2. npm 脚手架 CLI

### 2.1 命令体系

```bash
# 初始化新项目
npx lupine init <project-name>

# 更新现有项目
npx lupine update

# Skill 管理
lupine skill add <source> [options]   # 安装 Skill
lupine skill list                     # 查看已安装和推荐的 Skill
lupine skill remove <name>            # 移除 Skill

# 查看版本
npx lupine --version
```

### 2.2 init 命令

**功能**：
1. 创建 `.lupine/` 工作区目录
2. 从模板仓库复制基础文件（PRODUCT.md、rules/、specs/、AGENT.md 等）
3. 复制自研 Skill（builtin）到平台目录（`.opencode/skills/` 或 `.claude/skills/`）
4. 自动下载安装推荐的外部 Skill
5. 探查非推荐 Skill（风险告知）
6. 生成 `.lupineconfig.json`：
   ```json
   {
     "name": "<project-name>",
     "version": "0.1.0",
     "templateVersion": "x.y.z",
     "repoUrl": "...",
     "skills": {
       "installed": [...],
       "recommended": [...],
       "adopted": [...]
     }
   }
   ```
7. 生成 Agent 定义文件（自动注入 `available_skills`）

**交互流程**：
```
$ npx lupine init my-project
? 选择模板类型: [框架项目 / 应用项目 / 库项目]
✓ 已创建 .lupine/ 目录
✓ 已复制模板文件
✓ 已复制自研 Skill: lupine-diagram → .opencode/skills/lupine-diagram/
✓ 已安装推荐 Skill: brainstorming → .opencode/skills/brainstorming/
✓ 已安装推荐 Skill: impeccable → .opencode/skills/impeccable/
✓ 已生成 .lupineconfig.json

⚠️ 发现非推荐 Skill（可能影响 Agent 效果）
  ? unknown-skill        来源不明，未经验证
是否纳入 Lupine 配置?
  [Y] 纳入使用  [n] 保持忽略
```

### 2.3 update 命令

**功能**：
1. 读取 `.lupineconfig.json` 获取当前模板版本
2. 对比远程最新版本
3. 显示变更差异（新增/修改/删除的文件列表）
4. 用户确认后合并变更
5. **Skill 同步**：检测自研 Skill 和推荐外部 Skill 是否有更新
6. 更新 `.lupineconfig.json` 中的版本号

**选项**：
```bash
lupine update
lupine update --dry-run          # 预览变更
lupine update --force            # 强制覆盖用户修改
lupine update --sync-skills      # 同时更新 Skill
```

**保护机制**：
- 用户修改过的文件不会自动覆盖，会提示冲突
- 支持 `--dry-run` 预览变更
- 支持 `--force` 强制覆盖（慎用）
- `--sync-skills` 更新所有 builtin 和推荐外部 Skill（不处理用户 adopted 的 Skill）

### 2.4 Skill 管理命令

**`lupine skill add <source>`**

安装 Skill 到项目目录（`.opencode/skills/` 或 `.claude/skills/`）。

| 来源类型 | 示例 |
|---------|------|
| GitHub repo + 子目录 | `lupine skill add https://github.com/obra/superpowers --skill brainstorming` |
| GitHub repo（单 Skill）| `lupine skill add https://github.com/lupine-ai/skills-web-design` |
| npm 包 | `npx impeccable skills install` |
| 本地路径 | `lupine skill add ./local-skills/brainstorming` |

选项：`-s, --skill <name>`、`-p, --platform <name>`、`--dry-run`、`--force`

> **特定 Skill 安装命令**：部分 Skill（如 `impeccable`）使用自己的 npm 包安装（`npx impeccable skills install`），Lupine 在 Init 时自动调用。

**`lupine skill list`**

展示已安装、推荐未安装、非推荐探查三类 Skill。

**`lupine skill remove <name>`**

删除 Skill 目录并更新配置。

## 3. 模板同步与版本管理

### 3.1 模板仓库结构

```
lupine-templates/
├── base/                    ← 基础模板（所有项目通用）
│   ├── PRODUCT.md.template
│   ├── rules/
│   └── specs/
├── framework/               ← 框架项目模板
├── app/                     ← 应用项目模板
└── lib/                     ← 库项目模板
```

### 3.2 同步策略

**文件级同步**：
- 模板文件以 `.template` 结尾，支持变量替换（如 `{{projectName}}`）
- 非模板文件直接复制
- 删除检测：远程删除的文件，本地也标记为待删除

**冲突处理**：
- 如果本地文件被用户修改过，update 时提示冲突
- 用户可选择：保留本地 / 使用远程 / 手动合并

### 3.3 版本管理

- **单一版本源**：`packages/lupine/package.json` 的 `version` 字段为唯一版本源
- **本地记录**：`.lupineconfig.json` 的 `version` 字段记录初始化/上次更新时的 CLI 版本
- **版本检查**：`lupine update` / `lupine update --dry-run` 时：
  1. 读取本地 `.lupineconfig.json` 的 `version`
  2. 查询 npm registry (`https://registry.npmjs.org/lupine`) 获取 `dist-tags.latest`
  3. 与当前已安装 CLI 版本 (`package.json` 中的版本) 做 semver 比较
  4. 若 registry 版本更新，提示用户先升级 CLI (`npm install -g lupine@x.y.z`)
  5. 网络不通时优雅降级，不阻塞模板更新

## 4. 双渠道发布

### 4.1 npm Registry

**发布内容**：
- `lupine` CLI 包（包含 init/update/skill 命令）
- 模板文件（内嵌在 `templates/` 中，随包分发）

**版本策略**：
- 遵循语义化版本（SemVer）
- 每次发布更新 `package.json` 版本号
- 发布前运行测试套件

### 4.2 GitHub Release Tarball

**发布内容**：
- `lupine-templates-vX.Y.Z.tar.gz`：CLI 包内 `templates/` 目录的压缩包
- 供离线安装使用（企业内部网络无法访问 npm registry）
- 特定版本锁定（需要精确复现某个版本的模板）

**生成方式**：
```bash
cd packages/lupine
tar -czf "lupine-templates-${VERSION}.tar.gz" templates/
```

## 5. CI/CD 流程

### 5.1 自动化发布流水线

触发条件：`push tags v*`

```yaml
steps:
  1. Checkout
  2. Setup Node.js 22
  3. npm ci (packages/lupine/)
  4. npm test
  5. npm publish
  6. 打包 templates/ 为 tarball
  7. gh release create <tag> --generate-notes + 上传 tarball
```

### 5.2 GitHub Actions workflow

详见 `.github/workflows/release.yml`：
- `working-directory: packages/lupine` 确保在正确目录执行
- `NODE_AUTH_TOKEN` 从 GitHub Secrets 读取 npm token
- `GH_TOKEN` 使用默认 `secrets.GITHUB_TOKEN` 创建 Release
- 测试失败会阻塞 publish，publish 失败会阻塞 release

### 5.3 多仓库支持（v0.5 预留）

- 支持从多个模板仓库初始化项目
- `.lupineconfig.json` 支持 `templateRepo` 字段指定模板源
- 企业内部可搭建私有模板仓库
