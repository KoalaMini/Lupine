# Lupine 产品定位与愿景

> 本文档由分析器产出，经用户确认后锁定。作为所有功能需求的顶层锚点。

---

## 目标用户画像

- **用户群体**：使用 AI Coding 工具（Claude Code、Cursor、GitHub Copilot 等）的开发者、技术负责人，以及希望提升 AI 协作质量的团队
- **核心痛点**：
  - 当前 AI Coding 是"人给指令 → AI 写代码"的直给模式，质量看运气
  - 一个 session 里 AI 同时扮演 BA、架构师、开发，角色混杂
  - 需求假设、架构决策、代码实现缺少文档锚点，不可追溯
  - 无法让多个 AI 并行工作，效率受限
- **使用场景**：
  - 新项目启动：需要系统化地定义需求、设计架构、分模块并行实现
  - 已有项目加功能：希望在可控流程下引入新功能，确保质量和可追溯性
  - 团队 AI 协作：多 Agent 分工，减少人工重复沟通

## 核心价值主张

- **一句话描述**：Lupine 是一套多 Agent 协作开发流水线，将 AI Coding 从"独奏"升级为"交响"，让多个有明确角色的 Agent 协同工作，每个阶段有产出、有审查、可追溯。
- **与现有方案的区别**：
  - 现有工具（Cursor、Copilot）是单点增强，Lupine 是流程增强
  - 明确四角色分工（Analyzer / Planner / Executor / Evaluator），避免角色混杂
  - 两个评估门禁节点，确保需求→设计→代码的质量闭环
  - 所有产出物文档化（VISION / SPECS / PLANS / REVIEWS / TASKS）

## 成功标准

- **定性标准**：
  - 使用 Lupine 后，需求变更导致的返工显著减少
  - 技术方案与最终代码实现的一致性可验证
  - 团队成员能快速理解功能的设计意图和实现边界
- **定量标准**：
  - 从需求到可合并 PR 的平均周期缩短 ≥ 30%
  - 评估器门禁的通过率 ≥ 80%（首次通过）
  - 关键产出物（SPECS / PLANS / REVIEWS）的文档覆盖率达 100%

## 范围边界

### 明确列入范围
- 四角色（Analyzer / Planner / Executor / Evaluator）的定义与工作流规范
- 各阶段产出物的模板、命名规范与目录结构
- 评估门禁的 checklist 和审查流程
- 初始化脚本与项目骨架生成
- 与 Claude Code、opencode 等工具的集成指引
- 当前版本依赖人工启动 session；v2+ 探索自动化 Agent 调度与编排

### 明确不做（防止范围蔓延）
- 不替代 Git / CI/CD 基础设施（仅提供规范，不实现替代方案）
- 不做 AI 模型训练或微调（依赖底层模型的既有能力）
- 不做 UI 界面（当前阶段为 CLI + Markdown 文档驱动）
- 不替代代码编辑器的 AI 补全功能（与 Cursor / Copilot 互补而非竞争）
