# 进场体系与文档结构演进

> v0.7–v0.8 版本方案：基于 Harness Engineering 理念，重构 Lupine 的进场仪式和文档结构。

---

## 1. 背景与目标

### 1.1 问题

当前 Lupine 项目的进场仪式和文档结构存在几个核心问题：

- **CLAUDE.md 膨胀**：混合导航 + 规则 + 提示，Agent 上下文被不必要占用
- **FEATURES.json 四不像**：同时承担状态追踪、历史记录、文档索引三个职责，哪个都没做好
- **specs/ 兼管历史**：每个 spec 末尾的变更记录让文档不纯粹——spec 应该只描述"当前版本应该是什么样"
- **进场无协议**：Agent 新 session 开始时没有标准化流程，依赖隐式理解
- **缺少项目名片**：没有一处地方能让新进场的 Agent 快速了解"这是什么项目、用什么写的、怎么跑"

### 1.2 目标

> **Agent 进场时，从文件直接知道上下文，而不需要猜。**

具体来说：

1. **架构层**：Agent 5 秒内知道"这是什么项目、用什么写的、怎么跑"
2. **需求层**：Agent 知道"哪些功能已完成/进行中，当前要做什么"
3. **执行层**：Agent 知道"最近改了啥，下一步该干什么"

### 1.3 原则

- **specs = 当前状态**。不夹带历史。历史归 `git log` + `CHANGELOG.md`
- **版本先行**。一个版本更新多个 specs，确保每个 spec 反映目标版本的状态
- **不引入新复杂度**。不要 PROGRESS.md，不要 sessionLog，不要 reviews/latest
- **AI 可读优先**。FEATURES.json 保持 JSON 格式，AGENT.md 精简为导航目录

## 2. 三层进场体系

### 2.1 架构层（项目名片）

| 信息 | 文件 | 读一次就不重读 |
|------|------|--------------|
| 产品定位（一句话） | ARCHITECTURE.md "产品架构"段 | ✓ |
| 技术栈清单 | ARCHITECTURE.md "技术架构"段 | ✓ |
| 启动/测试/部署命令 | ARCHITECTURE.md "基础设施"段 | ✓ |
| 文档地图 | AGENT.md（已精简为导航目录） | ✗ session 必读 |

**ARCHITECTURE.md** 由 `lupine init` 时主动探查源码区自动生成。三段式，不超过 200 行。

### 2.2 需求层（功能全景 + 当前需求）

| 信息 | 文件 | 信息量 |
|------|------|--------|
| 有哪些功能、各自状态 | FEATURES.json（简化版） | ～20 行 |
| 当前需求要做什么 | specs/（按类别） | 一个 spec 50-200 行 |

*注：当前版本要做什么由**人指定**（"这版做 X"），Lupine 据此更新 specs。FEATURES.json 的 `status` 只做状态记录，不作为调度依据。*

**FEATURES.json 简化**：去掉 `history` 数组、`assetsRef` 字段、`currentSpec` 字段（改为 `spec`）。每个 feature 只保留 `id`、`name`、`status`、`spec`。

### 2.3 执行层（状态恢复）

| 信息 | 来源 | 信息量 |
|------|------|--------|
| 下一步该做什么 | plans/（已有，不改） | 一个 plan 30-80 行 |
| 最近改了啥 | `git log --oneline -20` | 20 行 |
| 当前要不要改什么 | `git status` | 几行 |

执行层**不新增加任何文件**。只做两件事：
1. 在编码规范中明确 commit 粒度原则——**可回退**
2. Agent prompt 指引 session 开始时自动读取上述信息

## 3. 文档结构变更

### 3.1 文件总览

| 文件 | 动作 | 原因 |
|------|------|------|
| `CLAUDE.md` | **更名 → AGENT.md，同时精简** | 去品牌化 + 纯导航目录 |
| `ARCHITECTURE.md` | **新增** | 架构层项目名片 |
| `CHANGELOG.md` | **新增** | 版本维度变更记录 |
| `FEATURES.json` | **简化** | 去掉 history 等历史负担 |
| 各 `specs/*.md` | **删除末尾变更记录** | specs = 当前状态原则 |
| `rules/coding.md` | **增强** | 增加 commit 质量指引 |

### 3.2 不变的文件

- `plans/`、`reviews/`、`proposal/`
- `.lupineconfig.json`
- `rules/git.md`、`rules/evals.md`
- Agent prompt 模板（仅微调）
- `.sketch/`

### 3.3 AGENT.md 内容模板

```markdown
# Lupine · AI 索引入口

## 项目名片
- **产品**：详见 [ARCHITECTURE.md](./ARCHITECTURE.md)
- **功能状态**：[FEATURES.json](./FEATURES.json)
- **变更历史**：[CHANGELOG.md](./CHANGELOG.md)

## 文档地图
| 目的 | 位置 |
|------|------|
| 需求规格 | `specs/`（按类别） |
| 执行计划 | `plans/` |
| 审查记录 | `reviews/` |
| 编码规范 | `rules/coding.md` |
| Git 规范 | `rules/git.md` |

## 进场协议
每个 session 开始时依序执行：
1. 读取 AGENT.md——了解文档地图
2. 读取 ARCHITECTURE.md——了解项目全貌
3. 读取 FEATURES.json——了解功能状态
4. 读取 `git log --oneline -20`——确认代码状态
5. 读取当前 `plans/`——确认下一步

## 进场指示灯
执行完上述协议后，输出一行状态摘要（不超过 3 行）：
```
📋 进场：架构层 [项目名] · 需求层 [当前功能] ([状态]) · 执行层 [最近提交摘要]
```
```

> 控制在 60 行左右。运行时始终注入。

### 3.4 ARCHITECTURE.md 内容模板

```markdown
# 项目架构

## 产品架构
Lupine 是一个多 Agent 协作开发框架。核心是一个"产品经理"角色的主 Agent
（Lupine），它带领四个专业子 Agent（探索器、规划器、执行器、评估器）协同
完成开发任务。

核心理念：从**独奏（Solo）**到**交响（Symphony）**。

## 技术架构
- 语言：Node.js 20+
- 框架：无（CLI 工具）
- 关键库：commander（CLI）、chalk（输出）
- 测试：Vitest
- 包管理：npm workspaces（monorepo）
- 分发：npm registry + tarball

## 目录结构
```
packages/lupine/
├── src/         ← 核心源码
├── templates/   ← 模板文件（Agent prompts、init 模板）
└── tests/       ← 测试
.lupine/         ← AI 工作区（配置 + 产出物）
```

## 基础设施
- 启动：`npm run dev`（端口自选）
- 测试：`npm test`（Vitest）
- 构建：`npm run build`
- 发布：`npm run release`
- CI：GitHub Actions（`.github/workflows/`）
```

> `lupine init` 时主动探查自动生成。

### 3.5 FEATURES.json 简化前后对比

**简化前**（v0.6）：
```json
{
  "id": "F3",
  "name": "上下文体系",
  "status": "completed",
  "description": "...",
  "currentSpec": "specs/上下文体系.md",
  "assetsRef": "specs/上下文体系.assets/",
  "history": [
    { "version": "0.6", "change": "...", "status": "completed" },
    { "version": "0.5", "change": "...", "status": "completed" },
    { "version": "0.3", "change": "...", "status": "superseded" },
    { "version": "0.1", "change": "...", "status": "superseded" }
  ]
}
```

**简化后**（v0.7）：
```json
{
  "id": "F3",
  "name": "上下文体系",
  "status": "completed",
  "spec": "specs/上下文体系.md"
}
```

### 3.6 specs/ 移除变更记录

当前每个 spec 末尾有一段变更记录表，如：

```markdown
## 变更记录
| v0.6 | 配置架构重构 | in-progress |
| v0.5 | 引入 FEATURES.json | completed |
```

**移除**。历史信息去向：
- `git log`——谁在什么时候改了什么
- `CHANGELOG.md`——版本维度汇总

spec 只描述当前版本"应该是什么样子"。

## 5. 不做的事

明确排除在本次演进之外：

- **PROGRESS.md**——Lupine 场景不需要跨 session 进度日志
- **plan 内嵌 sessionLog**——保持 plans/ 格式不变
- **reviews/latest 快捷引用**——不引入新复杂度
- **FEATURES.json 加验收标准**——不学 Anthropic 的 feature_list，保持简洁
- **specs/index.md**——共 4 个 spec 文件，不需要额外导航索引
- **rules/ 分层**——规则文件全部在 prompt 中内联，不做文件级分层
- **Agent prompt 大改**——只加一条"执行进场协议"指引
- **架构约束自动化 linter**——这是一个独立方向，不入本次演进

## 6. 进场协议全文（AGENT.md "进场协议"段落的展开版）

以下协议描述的是 Agent 每个 session 开始时的标准化动作：

```markdown
## 进场协议

每次 session 开始时按以下顺序恢复上下文：

### 架构层（20 秒）
1. 读取 AGENT.md——确认文档地图
2. 读取 ARCHITECTURE.md——确认项目全貌
3. 如果之前读过且架构未变，跳过此层

### 需求层（30 秒）
4. 读取 FEATURES.json——确认功能状态
5. 如果需要深入当前需求，读取对应 spec

### 执行层（10 秒）
6. 执行 `git log --oneline -20`——确认最近变更
7. 执行 `git status`——确认工作区状态
8. 读取当前 plans/——确认下一步动作

### 进场指示灯
执行完协议后，输出一条简短摘要（不超过 3 行）：
```
📋 进场：架构层 [项目名] · 需求层 [当前功能名] ([状态]) · 执行层 [最近提交摘要]
```

这个协议对 AI 的具体帮助：

| 步骤 | 帮 AI 避免了什么 |
|------|----------------|
| 读 AGENT.md | 花时间乱翻目录找文档 |
| 读 ARCHITECTURE.md | 用错技术栈、在错误位置改代码 |
| 读 FEATURES.json | 重复造已存在的功能，或漏实现 |
| 读对应 spec | 需求理解偏差导致返工 |
| git log + status | 基于过时状态工作导致冲突 |
| 读 plans/ | 不知道从哪开始 |
