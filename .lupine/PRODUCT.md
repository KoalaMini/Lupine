# Lupine 产品定义

> **From Solo to Symphony —— 从独奏到交响**

## 产品定位

Lupine 是一个多 Agent 协作开发框架。核心是一个"产品经理"角色的主Agent（Lupine），它不自己干所有活——它带领四个专业子Agent（探索器、规划器、执行器、评估器）协同完成开发任务。

## Why This Exists · 为什么需要这个

当前 AI Coding 工具（Cursor、Claude Code、GitHub Copilot 等）的使用方式本质上是 **人给指令 → AI 写代码** 的直给模式。问题在于：

- **质量看运气**——取决于 prompt 质量，缺少需求澄清和审查环节
- **角色混在一起**——一个 session 里既当 BA 又当架构师又当开发
- **不可追溯**——需求假设、架构决策、代码实现没有文档锚点
- **单线程**——无法让多个 AI 并行工作

这套架构将 AI Coding 从**独奏（Solo）** 变为**交响（Symphony）**：多个有明确角色分工的 Agent 协同工作，每个阶段有产出、有审查、可追溯。

## 核心理念

- **一个入口**：你只需跟 Lupine 对话，不需要在不同角色间手动切换
- **思考伙伴**：Lupine 不只是被动接收需求，它会挑战假设、带来新角度、带你发散再收敛，帮你找到"现阶段最适合你的方案"
- **专业分工**：四个子Agent各司其职，上下文隔离，工具限域
- **质量闭环**：评估器作为门禁，不合格的打回重做

## 架构

```
你 ──→ Lupine（调度器 / 产品经理）
          ├── 探索器  — 调研 （只读）
          ├── 规划器  — 写 plan（只读产出物）
          ├── 执行器  — 写代码、写测试
          └── 评估器  — 按约束审查（门禁控制）
          
每个 Agent 挂载 MCP + Skill + 约束（推荐制，可自定义）
```

## 角色职责

| 角色 | 类型 | 职责 | 产出物 | 能力层 |
|------|------|------|--------|--------|
| Lupine | 主Agent | 调度器 + 产品经理，对话探讨、写 specs、调度子 Agent | PRODUCT.md + specs/ | 调度决策 |
| 探索器 Explorer | 子Agent | 代码库探索、Web搜索、文档阅读（只读） | 情报摘要 | MCP + Skill |
| 规划器 Planner | 子Agent | 基于需求写执行计划（新功能/优化/Bug修复） | plans/ | MCP + Skill + 约束 |
| 执行器 Executor | 子Agent | 按 plan 写代码、写测试 | code + tests | MCP + Skill + 约束 |
| 评估器 Evaluator | 子Agent | 按约束审 plan + 审 code，门禁控制 | reviews/ | MCP + Skill + 约束 |

## 目标用户

使用 AI Coding 工具（Claude Code、opencode 等）的开发者和技术负责人，希望从"人给指令→AI 写代码"的直给模式，升级为有角色分工、质量门禁、可追溯的团队协作模式。

## 成功标准

- 从需求到可合并 PR 的平均周期缩短 ≥ 30%
- 评估器门禁首次通过率 ≥ 80%
- 关键产出物（specs / plans / reviews）文档覆盖率达 100%
- 需求变更导致的返工显著减少
- 技术方案与最终代码实现的一致性可验证

## 范围边界

### 列入范围
- Lupine 主Agent 的定义与工作流规范
- 五角色（Lupine/探索器/规划器/执行器/评估器）的定义与能力模型
- 产出物模型：PRODUCT.md + specs/ + plans/ + reviews/
- 约束机制（Agent prompt 直接内联约束）
- MCP/Skill 依赖管理与推荐清单
- 灵活调度：Lupine 自主决策，仅约束角色边界
- 与 Claude Code、opencode 等工具的集成

### 不做
- 不替代 Git / CI/CD 基础设施
- 不做 AI 模型训练或微调
- 不做 UI 界面（当前为 CLI + Markdown 文档驱动）
- 不替代代码编辑器的 AI 补全功能
- 不预设 Agent 调度路径（Lupine 自主决策）
- 不做独立评估引擎（约束注入 Agent prompt，非独立执行程序）

## 场景选择

| 场景 | 流程 |
|------|------|
| 新项目第一个功能 | 完整四角色流水线，含 PRODUCT.md 产出 |
| 已有项目加功能 | 完整流水线 |
| 紧急线上 bug | 跳过分析+规划，直接执行+审查，事后补 specs/ |
| 改文案/文档 | 直接执行后审查 |

## 设计理念

1. **需求可追溯**——每个功能的代码都能追溯到确认过的需规
2. **质量可评估**——规划和代码两个阶段设置评估门禁
3. **并行可执行**——不耦合的模块可多 Agent 并行实现
4. **知识可持续**——项目约定、质量标准、架构决策文档化
5. **人类最终把关**——AI 提供效率，人工决定质量
