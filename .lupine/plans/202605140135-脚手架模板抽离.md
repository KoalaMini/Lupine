## 1. 内容变更：rules/git.md 分支策略

### 1.1 框架自身 `rules/git.md`

将分支策略部分替换为：

```markdown
## 分支策略
- `master`: 保护分支，仅接受 release 的 PR；合入后打 tag 发布
- `dev`: 日常开发集成分支
- `release`: 从 dev 拉出，测试完成后 PR → master
- `feat/{功能名}`: 从 dev 拉出，开发完成后 PR → dev（功能名优先使用中文）
- `fix/{修复名}`: 从 dev 拉出，修复完成后 PR → dev（修复名优先使用中文）
```

### 1.2 模板 `assets/rules/git.md`

内容与 1.1 完全一致。作为新项目的默认模板，当用户接受 Git Flow 时直接复制。

### 1.3 空占位版本（用户拒绝时）

```markdown
# Git 协作规范

（此项目未采用 Lupine 推荐的 Git Flow，请根据团队规范自行补充）
```

## 2. 流程变更：bin/lupine-init 新增 Step 4.5

### 更新后的执行流程

```
Step 1-4: 同 v1.0（参数解析 → 路径解析 → 前置检查 → 复制模板）
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Step 4.5: 交互式 Git Flow 确认【新增】           │
│   read -p "是否采用 Lupine 推荐的                │
│            Git Flow 分支策略？(Y/n) " choice      │
│   ┌─ [Nn]* → 将 rules/git.md 替换为空占位文件    │
│   └─ 其他  → 保留已复制的 Git Flow 模板           │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
Step 5-6: 同 v1.0（后处理 → 输出结果）
```

### 代码变更（bin/lupine-init）

在 `cp -r "$ASSETS_DIR/"* "$INIT_DIR/"` 之后、skills 安装之前插入：

```bash
# Step 4.5: 交互式 Git Flow 确认
echo ""
echo "🐺 Git 分支策略"
read -p "  是否采用 Lupine 推荐的 Git Flow 分支策略（master/dev/release/feat/fix）？(Y/n) " choice
case "$choice" in
  [Nn]*)
    cat > "$INIT_DIR/rules/git.md" <<'GITEOF'
# Git 协作规范

（此项目未采用 Lupine 推荐的 Git Flow，请根据团队规范自行补充）
GITEOF
    echo "  已跳过 Git Flow 配置，rules/git.md 已置空"
    ;;
  *)
    echo "  已应用 Git Flow 分支策略"
    ;;
esac
```
