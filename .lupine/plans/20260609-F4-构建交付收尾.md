## 背景

`specs/构建交付.md` 中定义的构建交付模块包含：

---

## 设计决策

### 版本管理

- **单一版本源**：`packages/lupine/package.json` 的 `version` 字段
- **本地记录**：`.lupineconfig.json` 的 `version` 字段，在 `lupine init` 时写入当前 CLI 版本
- **检查方式**：`lupine update` / `lupine update --dry-run` 调用 `https://registry.npmjs.org/lupine`，读取 `dist-tags.latest`，与本地的 `.lupineconfig.json.version` 做 semver 比较
- **更新策略**：若 registry 版本更新，提示用户先升级 CLI（`npm install -g lupine@latest`），再用新 CLI 执行 `lupine update`

### CI/CD 流水线

触发条件：`push tags v*`

步骤：
1. `npm ci`（`working-directory: packages/lupine`）
2. `npm test`
3. `npm publish`
4. `gh release create <tag> --generate-notes` + 上传 `lupine-templates-<version>.tar.gz`

### 双渠道发布

- **npm Registry**：`npm publish` 发布 CLI 包
- **GitHub Release**：tarball 包含 `packages/lupine/templates/` 全部内容，供离线/企业内网安装

---

## 风险与回退

- npm registry 网络不通时，version check 应优雅降级（提示无法检查），不阻塞 `lupine update`
- GitHub Release 失败不应阻塞 npm publish（`continue-on-error: true` 或拆 job）
- 首次 npm publish 若包名被占用，workflow 会失败，需要人工处理
