## 二、核心架构对比

### 2.1 设计理念

| 维度 | Lupine | Superpowers |
|------|--------|-------------|
| **核心隐喻** | 交响乐队（Solo → Symphony） | 超能力（Superpowers） |
| **驱动力** | 四角色（Analyzer / Planner / Executor / Evaluator） | 14 个可组合 Skill |
| **触发方式** | 人工切换 Skill（`/lupine-*`） | **自动触发**（Agent 检测到上下文即激活） |
| **产出物** | Markdown 文档（VISION / SPECS / PLANS / REVIEWS / TASKS） | 设计文档 + 代码 + 测试（过程内嵌） |
| **审查节点** | 2 个（Plan 审查 + 代码审查） | 多阶段（设计确认 + 任务间 Review + 两级 Subagent Review） |
| **并行能力** | 多 Worktree 并行 Executor | Subagent 并行调度 |

### 2.2 工作流程对比

```
Lupine 流程（人工驱动）：
  用户 idea
    → Analyzer（需求澄清 → VISION + SPECS）人工确认
    → Planner（技术设计 → PLANS）
    → Evaluator ①（Plan 门禁）
    → Executor（并行实现 → 代码 + TASKS）
    → Evaluator ②（代码门禁）
    → 人工 Merge

Superpowers 流程（自动驱动）：
  用户 idea
    → brainstorming（自动激活，需求澄清 → 设计文档）人工确认分块
    → using-git-worktrees（自动创建隔离分支）
    → writing-plans（拆分为 2-5 分钟粒度的任务）
    → subagent-driven-development / executing-plans（子 Agent 执行 + 两级 Review）
    → test-driven-development（RED-GREEN-REFACTOR）
    → requesting-code-review（任务间审查，Critical 阻断）
    → finishing-a-development-branch（测试验证 → Merge/PR/Discard 选项）
```

### 2.3 关键差异点

| 差异 | Lupine | Superpowers |
|------|--------|-------------|
| **启动方式** | 用户主动调用 `/lupine-*` 切换角色 | Agent 自动检测并激活对应 Skill |
| **需求确认** | 分析器与用户多轮对话，确认后锁定 SPECS | Brainstorming 阶段分块展示设计，用户分块确认 |
| **Plan 颗粒度** | 一次性完整技术方案 | 拆分为 2-5 分钟可执行任务，含精确文件路径和验证步骤 |
| **执行模式** | Executor 单角色执行，多 Worktree 并行 | Subagent  per task，两级审查（spec compliance → code quality） |
| **测试策略** | 由 Executor 实现，Evaluators 审查 | 内嵌 TDD Skill，RED-GREEN-REFACTOR 强制循环 |
| **代码审查** | 1 个代码审查门禁（对照 SPECS + PLANS + EVALS） | 任务间持续审查 + 两级 Subagent Review |
| **文档化** | **强文档驱动**，所有产出物结构化归档 | 设计文档产出，但过程文档（Review/TASKS）内嵌于执行流 |
| **多平台** | 当前面向 Claude Code，支持 opencode | **8+ 平台**：Claude Code、Cursor、Codex、OpenCode、Copilot、Gemini、Factory Droid、Pi |
| **可扩展性** | 通过 `rules/` 规范扩展 | 通过 `writing-skills` Skill 自我扩展 Skill 库 |

## 三、Superpowers 详细拆解

### 3.1 14 个核心 Skill

| Skill | 作用 | 对应 Lupine 角色 |
|-------|------|-----------------|
| `brainstorming` | 需求澄清、设计细化、分块确认 | Analyzer |
| `writing-plans` | 技术方案拆分为可执行任务 | Planner |
| `executing-plans` / `subagent-driven-development` | 任务执行与子 Agent 调度 | Executor |
| `requesting-code-review` | 任务间代码审查 | Evaluator |
| `test-driven-development` | RED-GREEN-REFACTOR 强制 TDD | Executor + Evaluator |
| `systematic-debugging` | 4 阶段根因分析 | Evaluator |
| `using-git-worktrees` | 隔离分支并行开发 | Executor |
| `dispatching-parallel-agents` | 并发子 Agent 调度 | Executor |
| `finishing-a-development-branch` | 分支收尾（Merge/PR/Discard） | Evaluator |
| `verification-before-completion` | 完成前验证 | Evaluator |
| `receiving-code-review` | 响应代码审查反馈 | Executor |
| `using-superpowers` | Skill 系统介绍 | — |
| `writing-skills` | 创建新 Skill（元能力） | — |

### 3.2 四大哲学原则

1. **Test-Driven Development** — 永远先写测试
2. **Systematic over ad-hoc** — 流程优于猜测
3. **Complexity reduction** — 简洁是首要目标
4. **Evidence over claims** — 验证后再宣告成功

### 3.3 多平台插件架构

Superpowers 为每个平台提供原生插件：

- `.claude-plugin/` — Claude Code 官方/社区市场

- `.cursor-plugin/` — Cursor Agent

## 四、竞品优势（Superpowers 强于 Lupine 之处）

### 4.1 自动化程度更高

- **自动触发**：无需用户记忆和调用指令，Agent 检测上下文即激活对应 Skill
- **子 Agent 驱动**：每个任务 dispatch 新鲜 Subagent，避免上下文漂移，可连续自主运行数小时
- **自我扩展**：`writing-skills` Skill 让框架能自动生成新 Skill

### 4.2 测试纪律更强

- 内嵌 **RED-GREEN-REFACTOR** 强制循环，删除测试前编写的代码
- TDD 作为一等公民，而非执行器自行决定

### 4.3 平台覆盖更广

- 8+ 主流 AI Coding 工具原生支持
- 社区已有多平台移植

### 4.4 社区与影响力

- **19万+ Stars**，GitHub 上 AI Coding 工作流领域最热门项目
- 活跃的 Discord 社区和 Prime Radiant 组织背书
- 大量社区衍生项目和插件

### 4.5 任务颗粒度更细

- Plan 拆分为 **2-5 分钟**粒度的任务，精确到文件路径
- 每个任务面向"品味差、无判断力的初级工程师"编写，降低执行误差

## 五、竞品劣势（Lupine 可差异化之处）

### 5.1 文档化与可追溯性

| Superpowers | Lupine |
|-------------|--------|
| 设计文档产出，但 Review/TASKS 内嵌于执行流 | **结构化归档**所有产出物（VISION / SPECS / PLANS / REVIEWS / TASKS） |
| 过程文档依赖 Agent 会话历史 | 每个阶段产出物**持久化到文件系统**，人工随时可读 |

**差异化机会**：Lupine 的文档驱动更适合需要审计、合规、团队知识沉淀的场景。

### 5.2 角色明确性

| Superpowers | Lupine |
|-------------|--------|
| 14 个 Skill 按场景触发，角色边界由 Skill 定义 | **四角色严格分离**，每个角色有明确职责边界和禁止行为 |
| 一个 Agent 可能连续激活多个 Skill | Analyzer 绝不写代码，Executor 绝不决策，Evaluator 绝不实现 |

**差异化机会**：Lupine 的角色隔离更严格，适合需要明确责任归属的团队协作。

### 5.3 评估门禁的标准化

| Superpowers | Lupine |
|-------------|--------|
| Review 由 Subagent 执行，标准内嵌于 Skill | **独立 EVALS 规范文件**，门禁 checklist 可审计、可定制 |
| Critical issue 阻断进度 | PASS/FAIL 明确，评估报告结构化输出 |

**差异化机会**：Lupine 的评估标准外置为可配置规范，适合需要自定义质量门禁的团队。

### 5.4 人工介入设计

| Superpowers | Lupine |
|-------------|--------|
| 强调自动化，Agent 可自主运行数小时 | **分析器阶段强制人工确认**，所有需求变更需用户参与 |
| 设计分块确认，但流程以 Agent 为主导 | 人类是流程的驱动者，AI 是协作者 |

**差异化机会**：Lupine 更适合需要高度人机协作、需求频繁变更的场景。

### 5.5 范围边界清晰度

- Superpowers 功能持续扩展（14 Skills 且可自我增长）
- Lupine 明确列出**不做事项**（不替代 Git/CI、不做 UI、不做复杂 CLI）

**差异化机会**：Lupine 的明确边界宣言适合追求可控范围、避免工具蔓延的团队。

## 六、对 Lupine 的启示

### 6.1 可借鉴之处

1. **自动化触发机制**：考虑为 Lupine 增加上下文检测，在特定场景下主动提示切换角色
2. **TDD 强制循环**：在 Executor 规范中增加 RED-GREEN-REFACTOR 要求
3. **Subagent 调度**：Executor 阶段可探索为每个 Task 启动独立 Subagent
4. **多平台支持**：评估将 Lupine Skills 移植到 Cursor、Codex 等平台
5. **任务颗粒度**：PLANS 阶段进一步细化任务，精确到文件路径和验证步骤

### 6.2 应保持的差异化

1. **文档驱动**：坚持所有产出物结构化归档，不内嵌于执行流
2. **四角色隔离**：严格角色边界是 Lupine 的核心价值
3. **人工确认节点**：分析器阶段的人类参与是不可妥协的
4. **评估标准外置**：EVALS 作为独立规范文件，保持可配置性
5. **明确不做宣言**：范围边界是 Lupine 的可信度来源

### 6.3 潜在风险

- Superpowers 已建立极强的社区势能（19万+ Stars），新入场者需明确差异化
- Superpowers 的自动触发+Subagent 模式可能是行业演进方向
- 多平台支持是扩大用户基数的关键

---

## 七、结论

**obra/superpowers** 是当前 AI Coding 工作流领域**事实上的标杆项目**，其自动化 Skill 触发、Subagent 驱动执行、强制 TDD 和多平台覆盖值得 Lupine 深入研究。

Lupine 的差异化定位应聚焦于：

> **"文档驱动、角色隔离、人机协作可控的多 Agent 开发流水线"**

即：在 Superpowers 强调"让 Agent 自主飞"的方向上，Lupine 选择"让人类始终握有方向盘"，通过严格的角色分离和结构化文档产出，满足对可追溯性、责任边界和过程审计有要求的团队。
