## 发现的问题

### 🔶 问题 A：Prompt 正文仍引用 `specs/`

**文件：** `framework/init/templates/agents/lupine.prompt`

有两处仍然写着 `specs/`：

```
5. **产出** — 写 PRODUCT.md（产品级）或 specs/（功能级）
1. 读取已有 specs/后，检查新需求与已有 spec 范围是否相交
```

由于 `specs/` 已更名为 `reqs/`，prompt 正文也应对应更新。这会影响到所有通过 `lupine-init` 生成的目标项目中 Agent 的 prompt 文本。

**建议：** 将 `lupine.prompt` 中的 `specs/` 改为 `reqs/`。

### 🔶 问题 B：脚本命名与 spec 不一致

**Spec 写的是** `framework/init/lupine-init.sh`
**实际实现** `framework/init/lupine-init`（Python 脚本，无扩展名）

功能上无影响，Python shebang 可靠。但 `bin/lupine-init` symlink 指向 `framework/init/lupine-init`，路径正确。

**建议：** 无必要修改。通知 spec 下次迭代更新名称为 `lupine-init` 即可。

### 🔶 问题 C：缺少测试

`lupine-init` 脚本（215 行 Python）没有单元测试。手动测试已覆盖 peer/embedded/additive/claude 四种场景，但无自动化回归保障。

**影响：低。** 脚本逻辑清晰（流程式、无复杂算法），未来改动风险较低。

**建议：** 后续可补一个简单的 `pytest` 测试，在 temp 目录运行 `lupine-init` 并断言文件结构。
