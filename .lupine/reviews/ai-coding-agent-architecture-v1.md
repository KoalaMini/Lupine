# Lupine: AI Coding Agent 协作架构设计 需求规格说明书

- 版本: v1.2
- 状态: 已确认
- 最后更新: 2026-05-13

---

## 1. 项目背景与目标

### 1.1 问题陈述

当前 AI Coding 工具（Cursor、Claude Code、GitHub Copilot 等）的使用方式主要是"人给一句话指令 → AI 写代码"的直给模式。这种方式存在以下问题：

- AI 产出质量不稳定，取决于 prompt 质量
- 缺少需求澄清环节，AI 经常做假设、猜需求
- 缺少评估/审查环节，低质量代码直接进入代码库
- 不同阶段（需求分析、架构设计、编码、审查）混在一起，一个大模型 session 里既当 BA 又当架构师又当开发

### 1.2 目标

设计一套多 Agent 角色协作的开发流水线，让 AI Coding 从"单次对话"升级为"结构化团队协作"，实现：

1. 需求可追溯：每个功能的代码都能追溯到确认过的需规
2. 质量可评估：在规划和代码两个阶段设置评估门禁
3. 并行可执行：不耦合的模块可多 Agent 并行实现
4. 知识可持续：项目约定、质量标准、架构决策文档化

### 1.3 适用范围

- 使用 opencode 或类似具备多 session 能力的 AI Coding Agent 工具
- 新项目启动或成熟项目新增功能
- 单人开发模式（一个人用多个 agent session）或多人团队模式

## 2. 角色定义

### 2.1 角色全景

| 角色 | 职责 | 方式 | 产出物 | 需用户参与 |
|------|------|------|--------|-----------|
| 分析器 Analyzer | 与用户对话、澄清产品愿景与需求、写需规 | 迭代式 | VISION.md + SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md | ✅ 每轮确认 |
| 规划器 Planner | 基于确认的需规做技术设计 | 一次性 | PLANS/YYYYMMDD-{功能名称}.md | ❌ |
| 评估器 Evaluator | 在两个门禁节点做审查 | 一次性 | REVIEWS/{功能名称}-v{版本号}-{YYYYMMDD}-{类型}.md | ❌ |
| 执行器 Executor | 拆 task、写代码、写测试 | 拆并行 | 代码 + 测试 + TASKS/YYYYMMDD-{功能名称}.md | ❌ |

### 2.2 分析器（Analyzer）

**核心原则**：只问需求问题，不做技术设计，不写代码。

工作流程：

**阶段 A：产品定位与愿景（项目级，仅需一次）**

若项目尚无 `VISION.md`，分析器需先与用户探讨：

1. 目标用户画像（为谁解决什么问题）
2. 核心价值主张（一句话描述）
3. 成功标准（如何衡量成功）
4. 范围边界（明确不做的事项）
5. 产出 `VISION.md`，用户确认后锁定

**阶段 B：功能级需求澄清（每个功能一次）**

1. 读取 `VISION.md`、项目已有上下文（`CLAUDE.md`、已有 `SPECS/`）
2. 向用户提出 3-5 个结构化问题，覆盖：
   - 功能范围（做什么、不做什么）
   - 安全与约束（密码规则、权限、合规）
   - 边界情况（错误处理、上限、异常场景）
   - 非功能要求（性能、可扩展性、部署方式）
3. 根据用户回答，草拟 `SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md`
4. 展示给用户确认
5. 用户提出修改意见 → 迭代 SPECS → 再确认
6. 用户说"可以了" → 锁定版本，结束

**迭代终止条件**：用户明确确认当前版本。

### 2.3 规划器（Planner）

**核心原则**：基于已确认的需规，不做需求假设。

工作流程：

1. 读取 SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md（已确认版本）和 CLAUDE.md
2. 输出技术设计方案，包含：
   - 数据模型（表结构、字段类型、索引、约束）
   - 模块划分与职责边界
   - 接口定义（每个模块的公开函数签名）
   - 数据流（完整请求链路的每一步）
   - Task 拆分清单
3. 一次性输出，不迭代

**约束**：
- 不允许修改需规中已确认的功能范围和约束
- 不允许引入需规中未要求的技术方案
- 如果发现需规有遗漏 → 打回分析器，而非自己补

### 2.4 评估器（Evaluator）

**核心原则**：需规是最高优先级参照，其次是项目约定，最后是通用质量标准。

共两个审查节点：

**节点①：Plan 审查**
- 对照：SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md + CLAUDE.md
- 检查项示例：
  - 需规中所有功能是否都被 plan 覆盖
  - 安全约束是否在 plan 中体现
  - 需规中是否有 plan 遗漏的内容
- 输出：REVIEWS/{功能名称}-v{版本号}-{YYYYMMDD}-plan.md

**节点②：代码审查**
- 对照：SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md + PLANS/YYYYMMDD-{功能名称}.md + CLAUDE.md + EVALS.md
- 检查项示例：
  - 每个 API/功能是否与需规一致
  - 实现是否偏离设计（PLANS/ 中的接口定义）
  - 安全门禁是否满足（硬编码、日志泄露、密码处理）
  - 测试覆盖率是否达标
  - 代码风格是否符合项目约定
- 输出：REVIEWS/{功能名称}-v{版本号}-{YYYYMMDD}-code.md

**审查结论分级**：

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| FAIL (阻断) | 不满足需规或存在安全问题 | 打回对应阶段，附审查报告 |
| WARN (建议) | 不满足项目约定或质量标准 | 建议修改，不强制阻断 |
| PASS (通过) | 全部满足 | 允许合并 |

### 2.5 执行器（Executor）

**核心原则**：不自己做架构决策，严格遵循 PLANS/。

工作流程：

1. 读取 PLANS/YYYYMMDD-{功能名称}.md
2. 创建 TASKS/YYYYMMDD-{功能名称}.md 跟踪进度
3. 按 Task 逐个实现
4. 每个 Task 写单元测试
5. 完成后更新 TASKS/YYYYMMDD-{功能名称}.md

**并行规则**：
- 不依赖的 Task 可开多个 opencode session 并行执行
- 有依赖关系的 Task 按序执行（如 model 层完成后 handler 层才能跑）

## 3. 流水线工作流

### 3.1 完整流程

```
用户需求
   │
   ▼
┌──────────────────────────────────────────────────┐
│  分析器（迭代）                                │
│  · 每次输出 SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md                    │
│  · 给用户确认 → 用户说改就改 → 直到"可以了"       │
└──────────────────────┬───────────────────────────┘
                       │ SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md v1.x（已确认）
                       ▼
┌──────────────────────────────────────────────────┐
│  规划器（一次性）                              │
│  · 读 SPECS + CLAUDE.md                           │
│  · 输出 PLANS/YYYYMMDD-{功能名称}.md + TASK 拆解            │
└──────────────────────┬───────────────────────────┘
                       │ PLANS/YYYYMMDD-{功能名称}.md
                       ▼
┌──────────────────────────────────────────────────┐
│  评估器① — Plan 审查                              │
│  · 对照 SPECS + CLAUDE.md                         │
└──────┬───────────────┬───────────────────────────┘
       │ FAIL          │ PASS
       ▼               ▼
     打回规划器     ┌──────────────────────────────┐
                     │  执行器                        │
                     │  · 拆 TASKS/YYYYMMDD-{功能名称}.md           │
                     │  · 多 session 并行实现         │
                     │  · 写测试                     │
                     └──────┬───────────────────────┘
                            │ 代码 + 测试
                            ▼
               ┌──────────────────────────────────────┐
               │  评估器② — 代码审查                   │
               │  · 对照 SPECS + PLANS + CLAUDE + EVALS│
               └──────┬───────────────┬───────────────┘
                      │ FAIL          │ PASS
                      ▼               ▼
                   打回执行器       git merge
```

### 3.2 流程规则

1. **分析器和规划器不可同时进行**——需求必须先确认才能做设计
2. **评估器②未通过** → 打回执行器，附审查报告作为上下文
3. **评估器①未通过** → 打回规划器，附审查报告作为上下文
4. **评估器①发现需规遗漏** → 打回分析器，而非让规划器补
5. **不允许跳过评估阶段直接合并**
6. **PR 必须经人工审核后方可合并**（AI 评估器审查结果供参考，不替代人工）

### 3.3 例外处理

| 场景 | 处理方式 |
|------|---------|
| 需求变更 | 回到分析器，更新 SPECS，标记新版本号 |
| 实现中发现 plan 不可行 | 回到规划器修 plan，再经评估器①审查 |
| 紧急修复（如线上 bug） | 跳过调研+规划，直接执行+评估②，事后补 SPECS |
| 微小变更（如改文案） | 跳过调研和规划，直接执行后评估② |

## 4. 文件矩阵

### 4.1 文件全景

```
project-root/
├── CLAUDE.md              # 项目宪法

│   ├── 用户认证-v1.0-20260513.md
│   └── ...

│   ├── 20260513-用户认证.md
│   └── ...
```

### 4.2 CLAUDE.md — 项目宪法

**定位**：AI 的第一上下文文件，所有 agent session 自动读取。

**内容要求**：

```markdown
# Lupine 项目约定

## 技术栈
- 语言/框架/数据库/基础设施
- 关键依赖版本

## 架构原则
- 分层约定（handler/service/model 等）
- 依赖方向
- 错误处理规范
- 日志规范

## 代码约定
- 命名规范
- 测试要求
- 注释要求

## API 设计规范
- RESTful 路径规则
- 请求/响应格式
- 认证鉴权方式

## 环境与部署
- 环境变量列表
- 构建命令
- 部署方式

## Git 协作规范
### 分支策略
- `main`: 保护分支，仅通过 PR 合并
- `feat/{feature}`: 从 main 拉出，PR → main
- `fix/{bug}`: 紧急修复分支

### Commit 规范（Linux 社区风格）
- 格式: `{type}({scope}): {description}`
- type: feat / fix / docs / refactor / test / chore
- scope: 模块名
- 正文说明改动原因（why），而非内容（what）
- 尾部附 AI 辅助信息：`AI: {model-name}`

### PR 流程
- PR 必须经人工审核后方可合并
- AI 评估器审查结果供参考，不替代人工
- 禁止 self-merge
```

### 4.3 AGENTS.md — Agent 集群配置

**定位**：告知 AI 整个开发流水线的角色分工和工作流规则。

**内容要求**：见第 2 章角色定义 + 第 3 章工作流规则。

### 4.4 EVALS.md — 评估门禁标准

**定位**：评估器对照的 checklist，也是开发者自检的参考。

**内容要求**（按项目阶段补充）：

```markdown
# Lupine 评估标准

## 安全门禁（阻断）
- [ ] 密码/密钥/token 绝不硬编码
- [ ] 敏感信息不出现在日志或响应中
- [ ] SQL 注入防护
- [ ] 认证中间件覆盖所有需鉴权的路由

## 质量门禁（阻断）
- [ ] 新增代码测试覆盖率 ≥ 80%
- [ ] 圈复杂度 ≤ 15
- [ ] 不引入新的 lint error

## 架构门禁（警告）
- [ ] handler 层不含业务逻辑
- [ ] service 层不含 HTTP 细节
- [ ] 依赖方向是否正确
```

### 4.5 VISION.md — 产品定位与愿景

**定位**：分析器产出的项目级文档，作为所有功能需求的顶层锚点。

**内容要求**：

```markdown
# {项目名称} 产品定位与愿景

## 目标用户画像
- 用户群体特征
- 核心痛点

## 核心价值主张
- 一句话描述（ elevator pitch ）
- 与现有方案的区别

## 成功标准
- 如何衡量项目/功能的成功（定性 + 定量）

## 范围边界
- 明确列入范围的事项
- 明确不做的事项（防止范围蔓延）

## 变更记录
- v1.0: 初始版本
```

### 4.6 SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md — 需求规格说明书

**定位**：用户确认过的需求锚点，后续所有阶段的最高参照标准。

**内容要求**：

```markdown
# {功能名称} 需求规格说明书 v{版本}

## 功能范围
- 功能列表（做 vs 不做）

## 输入输出定义
- 每个功能的输入、输出、错误场景

## 边界与约束
- 限制条件（长度、格式、数量上限等）
- 错误处理策略

## 安全要求
- 认证、鉴权、加密、审计等

## 非功能要求
- 性能、可扩展性、可测试性、可维护性

## 变更记录
- v1.0: 初始版本
- v1.1: 角色名统一为"X器"后缀（分析器/规划器/评估器/执行器）；标题"集群"→"协作"；新增 Git 协作规范（分支策略、commit 附 AI 型号、PR 人工审核）
- v1.2: 文档命名规范改为 `{功能名称}-v{版本号}-{YYYYMMDD}.md`，优先使用中文名，版本号标识迭代；新增 VISION.md 产品定位与愿景环节；增加 Claude Code 集成说明和 lupine-init 初始化脚本
```

### 4.7 PLANS/YYYYMMDD-{功能名称}.md — 技术设计方案

**定位**：规划器的一次性输出，指导执行器编码。

**内容要求**：

```markdown
# {功能名称} 技术设计方案

## 数据模型
- 表结构 / 数据结构
- 索引与约束

## 模块划分
- 模块列表与职责
- 每个模块的文件路径

## 接口定义
- 模块间接口的函数签名
- 路由定义

## 数据流
- 每个用户操作的完整调用链路
- 时序或步骤说明

## Task 拆分
- T1: {描述}
- T2: {描述}
（每个 task 独立可执行、有明确完成标准）
```

### 4.8 REVIEWS/{功能名称}-v{版本号}-{YYYYMMDD}-{类型}.md — 审查报告

**定位**：评估器的审查结论，打回时作为上下文传递给对应阶段。

**内容要求**：

```markdown
# {功能名称} 审查报告

## 审查范围
- 审查对象（plan / 代码）
- 对照文档列表

## 审查结果

### FAIL（阻断）
- [ ] {问题描述} — 违反 {文档名} 第 {条款}

### WARN（建议）
- [ ] {问题描述}

### PASS（通过）
- ✅ {功能项}

## 统计数据
- 阻断项: {n}
- 建议项: {n}
- 通过率: {n}%
```

### 4.9 TASKS/YYYYMMDD-{功能名称}.md — Task 跟踪

**定位**：执行器用来跟踪进度的清单。

```markdown
# {功能名称} Task 跟踪

| Task | 描述 | 依赖 | 状态 | 负责人 |
|------|------|------|------|--------|
| T1   | ...  | 无   | done | agent-1 |
| T2   | ...  | T1   | done | agent-1 |
| T3   | ...  | T1   | in_progress | agent-2 |
| T4   | ...  | T2,T3 | pending | - |
```

## 5. 工具集成

### 5.1 opencode 多 session 操作

分析器:

```bash
opencode -m claude-sonnet-4.6 -p "你是分析器..."
```

规划器:

```bash
opencode -m claude-sonnet-4.6 -p "你是规划器..."
```

执行器（并行）:

```bash
# Session A
opencode -m gpt-5.4-codex -p "实现 T2 model/user.go..."

# Session B
opencode -m claude-sonnet-4.6 -p "实现 T4 handler/auth.go..."
```

评估器:

```bash
opencode -m claude-sonnet-4.6 -p "你是评估器，审查..."
```

### 5.2 CI/CD 集成

PR 合并前自动触发评估器审查（供人工参考，不替代人工审核）：

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: AI Review
        run: |
          opencode run -m claude-sonnet-4.6 -p "
          你是评估器，审查这个 PR。
          读取 EVALS.md，逐条检查代码是否满足。
          输出 REVIEWS/pr-${{ github.event.number }}.md
          "
```

### 5.3 模型选型建议

| Agent 角色 | 推荐模型 | 理由 |
|-----------|---------|------|
| 分析器 | Claude Sonnet 4.6 | 对话质量高，善于追问 |
| 规划器 | Claude Sonnet 4.6 / GPT-5.4 | 架构推理强 |
| 执行器（代码生成） | GPT-5.4 Codex / Claude Sonnet 4.6 | 代码生成速度快 |
| 执行器（测试生成） | Claude Haiku / MiniMax M2.1 | 便宜，够用 |
| 评估器 | Claude Sonnet 4.6 | 审查精度要求高 |
| 本地开发（敏感项目） | GLM-4.7 / Qwen2.5-Coder (Ollama) | 数据不出环境 |

### 5.4 Claude Code 集成

Claude Code 原生支持 Plan Mode、Worktree 隔离和 Agent 并行，可直接映射 Lupine 四角色流水线：

#### 分析器

```bash
# 方式一：普通 session，迭代确认需求
claude "你是分析器，阅读 CLAUDE.md 和已有 SPECS/，开始需求澄清..."

# 方式二：针对复杂需求，使用 Plan Mode 做结构化探讨
claude /plan "分析器：基于 VISION.md 澄清功能级需求"
```

- 若项目无 `VISION.md`，先探讨产品定位与愿景，产出 `VISION.md`
- 再迭代产出 `SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md`

#### 规划器

```bash
# 使用 Plan Mode 一次性输出技术设计方案
claude /plan "规划器：基于 SPECS/{功能名称}-v{版本号}-{YYYYMMDD}.md 输出 PLANS"
```

约束：规划器阶段建议强制使用 Plan Mode，确保一次性完整输出，避免中途实现代码。

#### 执行器（并行）

```bash
# Session A：隔离工作树实现 Task 1
claude /worktree "执行器：在隔离工作树中实现 T1..."

# Session B：隔离工作树实现 Task 2
claude /worktree "执行器：在隔离工作树中实现 T2..."
```

利用 Claude Code 的 Worktree 机制，每个 Task 在独立工作树中开发，避免文件冲突，最后合并。

#### 评估器

```bash
claude "你是评估器，对照 SPECS + PLANS + CLAUDE + EVALS，审查..."
```

评估器使用普通 session，读取所有参照文档，输出 `REVIEWS/{功能名称}-v{版本号}-{YYYYMMDD}-{类型}.md`。

#### 初始化工具

项目提供 `bin/lupine-init` 脚本，一键初始化完整骨架：

```bash
curl -sSL https://raw.githubusercontent.com/{repo}/main/bin/lupine-init | bash
```

或本地使用：

```bash
bin/lupine-init
```

## 7. 附录

### 7.1 术语表

| 术语 | 定义 |
|------|------|
| Agent 集群 | 多个有角色分工的 Agent session 组成的协作组 |
| 需规 | 需求规格说明书，用户确认过的功能描述 |
| 门禁 | 评估器设置的审查节点，未通过则阻止进入下一阶段 |
| 阻断 (FAIL) | 违反需规或存在安全问题时评估器的结论等级 |
| Task | 执行器拆出的最小可执行工作单元 |
| 人工审核 | 由人类开发者对 PR 进行审查，AI 评估器结果仅作参考，不替代人工判断 |

### 7.2 与其他方法论的关系

| 方法论 | 对应关系 |
|--------|---------|
| 敏捷开发 (Scrum) | 分析器 = PO + BA，评估器 = QA，执行器 = Dev |
| 传统瀑布 | 调研→设计→编码→测试 的映射 |
| TDD | EVALS.md 的测试门禁在编码前写好的原则 |
| CI/CD | 评估器②可集成到 CI pipeline 作为自动门禁 |

### 7.3 已知局限

1. 分析器的迭代效率取决于用户的配合程度，用户回复慢或模糊会影响整体进度
2. 评估器依赖 LLM 的判断力，对于需要领域专家知识的审查，可能不如人工
3. 多 session 并行时，需要人工协调 file lock / merge conflict
4. 当前模型在长对话上下文中存在"迷失在中间"的问题，评估器②审查大 PR 时可能有遗漏
5. AGENTS.md 定义的规则依赖 AI 自主遵守，当前无强制机制保证 AI 不越权
