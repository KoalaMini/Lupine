## 6. ⚠️ 执行前必须处理的 3 项问题

### 问题 1（中等）：`specs/上下文体系.md` 第 42 行的历史引用冲突

**现状**：Plan 要求将 line 42 的 `CLAUDE.md` 改为 `AGENT.md`。

但 line 42 原文是：
```
- 前身：v0.6 及之前的 CLAUDE.md
```

这是一句**历史陈述**——v0.6 时这个文件确实叫 CLAUDE.md。盲目替换为：
```
- 前身：v0.6 及之前的 AGENT.md
```
在语义上是错误的（v0.6 时还没有 AGENT.md）。

**建议**：
- **方案 A**：将这句话改为 `- 前身：v0.6 及之前的 CLAUDE.md（详见 CHANGELOG.md）`，保持历史准确
- **方案 B**：将这句单独列入"不修改"列表（与 proposal/reviews/plans 同理，是历史引用）
- **方案 C**：完全移除该行——specs 遵循"当前状态"原则，不夹带历史

**推荐方案 A**，既保持了历史准确，又指引读者去 CHANGELOG.md 了解完整记录。T5 的验收条件也需要相应放宽为：与此类历史引用无关的 CLAUDE.md 已被替换。

### 问题 2（中等）：两个 `_manifest.json` 的 hash 计算需要区分

**现状**：Plan 第 151 行给出命令 `shasum -a 256 .lupine/AGENT.md`。

但涉及两个不同的 manifest：
1. `.lupine/_manifest.json` → hash of `.lupine/AGENT.md`（硬编码内容）
2. `packages/lupine/templates/_manifest.json` → hash of `packages/lupine/templates/AGENT.md`（含 `{project_name}` 占位符）

它们的内容**不同**（一个有实际项目名，一个有占位符），所以 hash 也不同。

**建议**：在 Step 5.5 和 Step 5.10 中分别给出 hash 计算命令，避免混淆：
- Step 5.5：`shasum -a 256 .lupine/AGENT.md`
- Step 5.10：`shasum -a 256 packages/lupine/templates/AGENT.md`

### 问题 3（低）：缺少 `{project_name}` 占位符的一致性检查

**现状**：Step 5.10 要求 templates/AGENT.md 使用 `{project_name}` 占位符，但验收条件是 grep 无 CLAUDE.md 匹配——没有检查占位符是否存在。

**建议**：为 Step 5.10 增加一条验收条件：
```bash
grep "{project_name}" packages/lupine/templates/AGENT.md
# 应返回匹配，确认占位符存在
```
