## 1. 架构与组件划分

本功能为纯结构调整，无运行时服务或复杂组件。核心架构只有两层：

| 组件 | 类型 | 职责 |
|------|------|------|
| `assets/` | 静态模板目录 | 存放完整的初始化模板文件，作为被复制源 |
| `bin/lupine-init` | Bash 脚本 | 接收参数 → 路径解析 → 前置检查 → 复制模板 → 后处理 → 输出结果 |

### 1.1 assets/ 目录结构

`assets/` 与目标项目的目录结构保持 1:1 映射，确保复制后无需任何转换：

```
assets/
├── CLAUDE.md              # AI 索引入口模板（占位符版本）
├── VISION.md              # 产品愿景模板（占位符版本）
├── rules/
│   ├── agents.md          # Agent 角色定义与工作流
│   ├── coding.md          # 技术栈与编码规范（占位符版本）
│   ├── evals.md           # 评估门禁 checklist
│   └── git.md             # Git 协作规范
├── specs/
│   └── README.md          # 规格说明书目录说明
├── plans/
│   └── README.md          # 设计方案目录说明
├── reviews/
│   └── README.md          # 审查报告目录说明
└── tasks/
    └── README.md          # 任务跟踪目录说明
```

### 1.2 脚本执行流程

```
用户执行: lupine-init [项目名] [目标目录]
    │
```

## 2. 数据流与接口定义

### 2.1 脚本接口

```bash
# 用法不变
lupine-init [项目名] [目标目录]

# 示例
lupine-init                    # 初始化到当前目录，项目名 my-project
lupine-init my-app ./projects  # 初始化到 ./projects，项目名 my-app
```

### 2.2 输入输出

| 输入 | 来源 | 说明 |
|------|------|------|
| 项目名 | CLI 参数 $1 | 仅用于输出提示，不写入文件 |
| 目标目录 | CLI 参数 $2 | 模板复制目的地 |
| assets/ 目录 | 框架安装路径 | 脚本通过相对路径自动定位 |

| 输出 | 目标 | 说明 |
|------|------|------|
| 复制后的文件树 | 目标目录 | 与 assets/ 结构完全一致 |
| 控制台输出 | stdout | 项目信息、文件列表、下一步提示 |

### 2.3 路径解析逻辑

```bash
# 关键代码片段
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
LUPINE_FRAMEWORK_DIR="$(cd "$SCRIPT_DIR/.." && pwd)"
ASSETS_DIR="$LUPINE_FRAMEWORK_DIR/assets"

# 检查 assets 存在
if [ ! -d "$ASSETS_DIR" ]; then
    echo "Error: 模板目录不存在: $ASSETS_DIR" >&2
    echo "请检查 Lupine 框架是否正确安装。" >&2
    exit 1
fi
```

## 3. 错误处理策略

| 场景 | 检测方式 | 处理方式 | 退出码 |
|------|---------|---------|--------|
| `assets/` 目录缺失 | `[ ! -d "$ASSETS_DIR" ]` | stderr 输出错误路径和提示 | 1 |
| 目标目录创建失败 | `mkdir -p` 失败（`set -e`） | bash 自动退出 | 1 |
| 复制失败 | `cp -r` 失败（`set -e`） | bash 自动退出 | 1 |
| 参数缺失 | 使用默认值 | 静默处理，不报错 | 0 |

**原则**：保持与现有 `lupine-init` 相同的错误处理风格——`set -e` 捕获意外错误，显式检查捕获预期错误（assets 缺失）。

## 5. 执行顺序

```
Step 1: 创建 assets/ 目录及所有模板文件
        └─ 从 lupine-init 的 heredoc 提取内容

Step 2: 重写 bin/lupine-init
        └─ 删除 heredoc，改为 cp -r assets/*

Step 3: 迁移 .lupine/ 内容到根目录
        └─ 移动所有 specs/plans/reviews/tasks/VISION.md

Step 4: 更新 CLAUDE.md
        └─ 移除 .lupine/ 引用，更新自迭代说明

Step 5: 更新 README.md
        └─ 更新文件结构说明，加入 assets/

Step 6: 删除 .lupine/ 目录

Step 7: 验证
        └─ 运行 lupine-init /tmp/test-project
        └─ 对比文件结构与预期一致
        └─ 对比与迁移前初始化结果一致
```

## 7. 关键设计决策

### 7.1 为何用 `cp -r` 而非 rsync/tar？

- `cp -r` 是 POSIX 标准命令，无需额外依赖
- 模板文件数量少（~10 个），无需增量同步
- 符合"简单可依赖"原则

### 7.2 为何 `assets/` 放在仓库根而非 `lib/assets`？

- 与 `bin/` 同级，脚本通过 `$(dirname "$0")/..` 即可定位
- 目录语义清晰：`assets/` = 模板资产，`bin/` = 可执行脚本

### 7.3 为何不保留 `.lupine/` 作为过渡？

- 用户明确确认"不需要向后兼容，现在处于 MVP 版本"
- 一次性完成迁移，避免后续维护两套路径的引用

## 8. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 迁移时遗漏 `.lupine/` 中的文件 | 中 | 中 | 迁移前列出所有 `.lupine/` 文件做 checklist |
| `lupine-init` 重写后路径解析错误 | 低 | 高 | 验证框架安装于不同目录（非当前目录）时的初始化 |
| 模板内容提取时丢失格式 | 低 | 中 | 复制后 diff 对比 heredoc 原始内容 |
| `CLAUDE.md`/`README.md` 更新遗漏 | 中 | 中 | grep 检查 `.lupine` 字符串残留 |
