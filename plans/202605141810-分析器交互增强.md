# 技术设计方案：分析器交互增强

> 对应 Spec：`specs/分析器交互增强-v1.0-202605141750.md`  
> 规划时间：2026-05-14 18:10  
> 规划器：Lupine Planner

---

## 1. 现状分析

当前 `lupine-analyzer/SKILL.md` 的交互模式为**批量提问 + 一次性产出**：

```
用户提出需求
    → Analyzer 一次提 3-5 个结构化问题
    → 用户一次性回答
    → Analyzer 草拟完整 specs
    → 用户确认/迭代
```

**问题：**
- 3-5 个问题同时抛出，用户认知负担高
- 无可视化辅助，复杂架构/流程仅靠文字描述
- 设计一次性展示，用户难以逐块确认
- 无多方案对比机制，用户缺少选择空间
- 无自检环节，spec 质量依赖一次性生成质量

---

## 2. 技术方案

### 2.1 总体架构

本次改动为**纯文档层重构**，不涉及代码/脚本。核心是对 `SKILL.md` 的工作流进行结构性重写。

```
┌─────────────────────────────────────────┐
│           lupine-analyzer/SKILL.md      │
│  ┌─────────────────────────────────┐    │
│  │  一问一答交互引擎（重写工作流）   │    │
│  │  - 规模评估 → 分解/直接进入      │    │
│  │  - 单问题循环（AskUserQuestion）  │    │
│  │  - 2-3 方案对比 + trade-offs     │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  Visual Companion（新增章节）    │    │
│  │  - SVG 生成规范                  │    │
│  │  - 文件路径管理                  │    │
│  │  - 多版本对比                    │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  分块确认设计（新增章节）         │    │
│  │  - Section-by-section approval   │    │
│  │  - 回退机制                      │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │  YAGNI + Spec 自检（新增章节）    │    │
│  │  - 写前功能必要性确认              │    │
│  │  - 写后四维度自检                 │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 2.2 关键决策

| 决策点 | 方案 A（选） | 方案 B（弃） | 理由 |
|--------|-------------|-------------|------|
| **交互触发** | 用户调用 `/lupine-analyzer` 后，Analyzer 自主按新流程执行 | 增加子 Skill（`/lupine-brainstorm`） | 避免技能碎片化，Analyzer 本身就是 brainstorming 角色 |
| **草图格式** | 内联 SVG（`<svg>` 标签嵌入 markdown） | 外部 HTML 文件 + 链接 | 无需 server，GitHub 原生渲染，符合 Lupine CLI + Markdown 哲学 |
| **草图生成** | Agent 对话中直接 Write SVG 文件 | 脚本自动化生成 | specs 阶段为对话驱动，无脚本介入必要 |
| **分块确认** | 对话中逐 section 确认，不持久化中间状态 | section 状态写入临时文件 | 中间状态由对话上下文自然承载，过度持久化增加复杂度 |
| **方案对比** | 终端表格 + 并排 SVG 图 | 浏览器 A/B 测试页 | 不引入 server，保持 CLI 体验 |

### 2.5 SVG 草图规范（技术细节）

**内联 SVG 嵌入 markdown 示例：**

```markdown
## 架构方案对比

### 方案 A
<svg xmlns="http://www.w3.org/2000/svg" width="400" height="200">
  <rect x="10" y="10" width="100" height="60" fill="#f0f0f0" stroke="#333" rx="4"/>
  <text x="60" y="45" text-anchor="middle" font-size="14">Analyzer</text>
  <line x1="110" y1="40" x2="180" y2="40" stroke="#333" marker-end="url(#arrow)"/>
  <rect x="180" y="10" width="100" height="60" fill="#f0f0f0" stroke="#333" rx="4"/>
  <text x="230" y="45" text-anchor="middle" font-size="14">Planner</text>
  <defs>
    <marker id="arrow" markerWidth="10" markerHeight="10" refX="9" refY="3" orient="auto">
      <path d="M0,0 L0,6 L9,3 z" fill="#333"/>
    </marker>
  </defs>
</svg>

### 方案 B
<svg xmlns="http://www.w3.org/2000/svg" width="400" height="200">
  <!-- ... -->
</svg>
```

**SVG 风格约束：**
- 配色：`#333`（主文字）、`#666`（次要）、`#f0f0f0`（填充）、`#e74c3c`（强调，仅用于关键节点）
- 字体：系统默认 sans-serif，不小于 12px
- 尺寸：宽度 ≤ 800px，高度自适应
- 无 `<script>`、无 `<foreignObject>`、无外部资源引用

**文件管理时序：**
```
对话中生成草图
    → 写入 .lupine/sketches/{功能名}/arch-v1.svg
    → 展示给用户（内联 SVG 或文件链接）
    → 用户确认后
    → 复制到 specs/attachments/{功能名}-v{版本}-{时间}/arch-final.svg
    → 在 specs 中引用附件路径
```

### 2.6 一问一答交互时序

```
Turn 1: 用户提出需求
Turn 2: Analyzer 评估规模
        └─ 过大 → "该项目涉及多个独立子系统，建议分解。以下是可以拆分的模块：... 我们先从哪个开始？"
        └─ 适中 → 进入澄清循环
Turn 3: Analyzer 问第 1 个问题（选择题）
        [AskUserQuestion: 问题 + 2-4 个选项]
Turn 4: 用户回答
Turn 5: Analyzer 问第 2 个问题（如需要追问）
        ...
Turn N: 澄清完成 → 提出 2-3 方案 + trade-offs + 草图
Turn N+1: 用户选择方案
Turn N+2: 进入分块确认
        "以下按 section 展示设计，请逐块确认："
        Section 1: 范围 → [确认/修改]
        Section 2: 架构 → [确认/修改]
        ...
Turn M: 全部确认 → YAGNI 自检 → 写入 spec
Turn M+1: Spec 自检 → 展示完整 spec → 用户最终确认 → 锁定
```

## 3. 边界与风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 一问一答轮次过多导致用户疲惫 | 中 | 高 | 设置上限：单功能最多 15 轮问答；复杂项目强制分解 |
| SVG 生成质量不稳定 | 中 | 中 | 降级策略：SVG 生成失败时自动降级为 ASCII 图或纯文本 |
| 分块确认增加交互轮次 | 高 | 中 | Section 合并策略：简单功能可合并为 2-3 个大块 |
| SKILL.md 过长导致 AI 上下文效率下降 | 中 | 中 | 保持 SKILL.md 精炼，将详细 SVG 模板移至独立参考文件 |
| 与现有 specs 产出不兼容 | 低 | 高 | 保持 specs 文件格式不变，仅改变交互流程 |
