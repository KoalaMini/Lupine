# 需求规格说明书：远程一键初始化（Release 构建版）

> **功能名称**：远程一键初始化（curl-install + Release 构建）
> **版本**：v1.1
> **日期**：2026-05-14 16:30
> **状态**：待用户确认

---

## 1. 功能概述

提供类似 `curl -fsSL https://raw.githubusercontent.com/.../install | bash` 的一键安装体验，让 Lupine 的上下文模块能够**直接注入到用户现有的项目中**。

**关键设计决策**：
- Release 产物是"干净的"成品，只包含用户需要的文件
- 通过自动化构建脚本过滤掉 Lupine 自身的项目记录
- 用户获取方式：GitHub Release tarball（一键下载）
- 安装脚本 `install.sh` 放在 master 分支，通过 GitHub raw 访问
- master 保持正常项目记录，不做干净分支

---

## 2. 用户旅程

### 方式一：curl | bash（推荐）
```bash
# 用户在自己的项目目录中执行
curl -fsSL https://raw.githubusercontent.com/koalamini/lupine/master/install.sh | bash

# 系统响应：
# 1. 查询 GitHub Release 获取最新版本
# 2. 下载 lupine-context-v{VERSION}.tar.gz
# 3. 解压并复制到当前目录
# 4. 替换占位符
# 5. 输出使用指引
```

### 方式二：手动下载
```bash
# 从 GitHub Release 页面手动下载 tarball，解压后执行
# 适合无法使用 curl 的环境
curl -fsSL https://github.com/koalamini/lupine/releases/latest/download/lupine-context.tar.gz | tar -xz
./lupine-context-*/bin/lupine-init
```

## 3. 功能范围

### 3.1 明确列入范围

| 功能点 | 说明 |
|--------|------|
| 构建脚本 | `scripts/build-release.sh`：过滤文件、生成干净 tarball |
| GitHub Actions 工作流 | `.github/workflows/release.yml`：tag push 时自动触发构建 |
| GitHub Release 自动创建 | 打包 tarball 并上传 Release assets |
| 远程安装脚本 | `install.sh`：放在 master 分支，通过 GitHub raw 访问 |
| 版本解析 | 默认获取 GitHub Release 最新版；支持 `VERSION=v1.2.0` 环境变量指定 |
| 资源下载 | 从 GitHub Release 下载 `lupine-context-v{VERSION}.tar.gz` |
| 嵌入安装 | 将 assets 解压到当前工作目录（非子目录），与现有文件合并 |
| 占位符替换 | 自动提取当前目录名替换 `assets/CLAUDE.md` 中的 `{项目名称}` |
| 冲突检测 | 若目标目录已存在 Lupine 文件（如 `rules/agents.md`），提示用户并询问是否覆盖 |
| 使用指引 | 安装完成后打印清晰的下一步操作说明 |

### 3.2 明确不做

| 功能点 | 说明 |
|--------|------|
| 不管理 Lupine 版本更新 | 安装脚本只负责首次嵌入，后续更新需用户手动执行 |
| 不替代包管理器 | 不发布到 npm、Homebrew、apt 等 |
| 不做全局安装 | 不安装到 `~/.lupine/` 或系统 PATH，只嵌入当前项目 |
| 不做交互式配置 | Git Flow 分支策略等改为默认值+文档说明 |
| 不保留 specs/plans 模板 | 用户项目不需要需求文档模板，只保留空目录（带 `.gitkeep`） |
| 不用外部 CDN | 安装脚本直接走 GitHub raw，不额外配置 CDN |
| 不维护常驻干净分支 | 成品只通过 GitHub Release tarball 分发，不用分支 |

## 4. 详细需求

### 4.1 构建脚本（`scripts/build-release.sh`）

**触发条件**：GitHub Actions 在 `v*` tag push 到 master 时自动调用

**输入**：当前仓库完整代码（含项目记录）
**输出**：
`lupine-context-v{VERSION}.tar.gz`（上传到 GitHub Release）

**构建流程**：
```bash
#!/bin/bash
set -e

VERSION="${1:-$(git describe --tags --always)}"
BUILD_DIR=$(mktemp -d)

echo "🏗️  构建 Lupine Release v$VERSION"

# 1. 创建干净的构建目录
mkdir -p "$BUILD_DIR/lupine-context-$VERSION"

# 2. 复制成品文件
cp -r bin "$BUILD_DIR/lupine-context-$VERSION/"
cp -r assets "$BUILD_DIR/lupine-context-$VERSION/"
cp -r scripts "$BUILD_DIR/lupine-context-$VERSION/"
cp -r skills "$BUILD_DIR/lupine-context-$VERSION/"
cp README.md "$BUILD_DIR/lupine-context-$VERSION/"

# 3. 清理不需要的文件（确保没有项目级文件混入）
#    即使源目录结构变化，这里也要显式排除
rm -f "$BUILD_DIR/lupine-context-$VERSION/assets/README.md" 2>/dev/null || true

# 4. 生成版本标记
echo "$VERSION" > "$BUILD_DIR/lupine-context-$VERSION/.lupine-version"

# 5. 打包
tar -czf "lupine-context-$VERSION.tar.gz" -C "$BUILD_DIR" "lupine-context-$VERSION"

# 6. 清理
rm -rf "$BUILD_DIR"

echo "✅ 构建完成: lupine-context-$VERSION.tar.gz"
```

**成品目录结构**：
```
lupine-context-v1.0.0/
├── bin/
│   └── lupine-init          # 初始化脚本
├── assets/

│   ├── VISION.md            # 模板：产品愿景

│       └── git.md           # 模板：Git 协作规范
```

### 4.2 GitHub Actions 工作流（`.github/workflows/release.yml`）

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  build-and-release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Release
        run: |
          VERSION=${GITHUB_REF#refs/tags/}
          ./scripts/build-release.sh "$VERSION"

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: lupine-context-*.tar.gz
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### 4.3 远程安装脚本（`install.sh`）

**托管位置**：master 分支根目录（通过 GitHub raw 访问）

```bash
# 默认行为：安装最新版到当前目录
curl -fsSL https://raw.githubusercontent.com/koalamini/lupine/master/install.sh | bash

# 指定版本
curl -fsSL https://raw.githubusercontent.com/koalamini/lupine/master/install.sh | VERSION=v1.2.0 bash

# 强制覆盖（非交互）
curl -fsSL https://raw.githubusercontent.com/koalamini/lupine/master/install.sh | FORCE=1 bash

# 仅下载不安装（调试用）
curl -fsSL https://raw.githubusercontent.com/koalamini/lupine/master/install.sh | DRY_RUN=1 bash
```

**执行流程**：
1. **环境检查** → curl、写权限
2. **版本解析** → GitHub API 取最新版（可降级处理限流）
3. **资源下载** → `lupine-context-v{VERSION}.tar.gz`
4. **冲突检测** → 已有文件时拒绝覆盖（需 `FORCE=1`）
5. **解压与嵌入** → 解压到当前目录
6. **占位符替换** → `{项目名称}` → `basename $PWD`
7. **输出指引** → 清晰的下一步操作说明

### 4.4 冲突检测逻辑

```bash
# 检测当前目录是否已有 Lupine 文件
for file in "rules/agents.md" "rules/evals.md" "CLAUDE.md"; do
    if [ -f "$file" ] && [ -z "$FORCE" ]; then
        echo "⚠️  检测到已有 Lupine 文件: $file"
        echo "    如需覆盖，请重新执行并添加 FORCE=1"
        exit 1
    fi
done
```

### 4.6 降级与错误处理

| 场景 | 行为 |
|------|------|
| GitHub API 限流（403） | 尝试直接访问 `latest` release 的已知 URL，失败则提示用户手动指定版本 |
| 下载 404 | 退出码 1，提示 "版本 {VERSION} 不存在" |
| 文件冲突（未加 FORCE） | 退出码 1，列出冲突文件，提示加 FORCE=1 |
| 网络超时 | 重试 3 次（间隔 2s），仍失败则退出 |

## 5. 安全与约束

- **最小权限**：脚本只操作当前目录
- **显式确认**：默认不覆盖已有文件
- **可审计**：执行前打印下载 URL 和版本号
- **临时文件**：下载内容先放 `/tmp`，完成后清理
