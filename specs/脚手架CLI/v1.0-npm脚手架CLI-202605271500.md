# npm 脚手架 CLI

> Lupine 官方 CLI 工具：项目初始化与框架更新
> 版本：v1.0
> 创建时间：2026-05-27

---

## 1. 功能概述

将 Lupine 框架的初始化工具从 Python 脚本 + GitHub Release tarball 模式，迁移为 **Node.js npm 包**。用户通过 `npx lupine init` 即可在当前目录创建 `.lupine/` 工作区，并通过 `lupine update` 同步框架更新。

### 核心变化（相对旧版）

| 维度 | 旧版（Python） | 新版（npm） |
|------|---------------|-------------|
| 运行时依赖 | Python 3 + PyYAML | Node.js 18+ |
| 安装方式 | `curl \| bash` + tarball | `npx lupine` 零安装 |
| 工作目录 | peer / embedded 两种模式 | 统一 `.lupine/` |
| 仓库关联 | 无 | 初始化时录入，支持多个 |
| 模板同步 | 手动维护 | 自动 `sync-templates.js` |
| 框架更新 | 无内置机制 | `lupine update` 命令 |

---

## 2. 目标用户

- 使用 Lupine 多 Agent 协作开发框架的开发者
- 需要在现有项目（或多个项目）旁搭建 AI 开发中枢的用户

## 3. 功能范围

### 列入范围

| 功能 | 说明 |
|------|------|
| `lupine init` | 交互式/非交互式初始化 .lupine/ 工作区 |
| `lupine update` | 基于 checksum 智能更新模板文件 |
| 多仓库支持 | 初始化时录入多个代码仓库路径 |
| ⚙️ 子命令体系 | commander 框架，预留扩展 |
| 📦 npm 发布 | 发布到公共 npm registry（@latest） |
| 🔄 模板自动同步 | sync-templates.js 从项目根规则编译模板 |
| 📄 .lupineconfig.json | 工作区配置文件（repo 路径、版本等） |
| 🤖 Agent 定义生成 | 根据平台生成 `.opencode/agents/` 或 `.claude/agents/` 4 个 Agent 定义文件 |

### 明确不做

| 功能 | 说明 |
|------|------|
| 不做 peer / embedded 模式区分 | 统一输出到 .lupine/ |
| 不做全局安装引导 | 推荐 npx，用户也可 npm i -g |
| 不替代 GitHub Release 分发 | tarball 保留，双渠道发布 |
| 不管理用户仓库的版本 | 仅记录路径，不介入用户仓库的 git |
| 不做 CI/CD 集成 | 仅初始化工具，不进入开发流水线 |
| 不做旧版迁移工具 | 旧版用户手动重新 init |
| 不自动检测 AI 平台 | 交互式让用户选择 opencode / claude |

### 4.2 `init` 命令详细实现

#### 用户旅程

```bash
$ npx lupine init

🐺 Lupine 项目初始化
--------------------

? 请输入项目名称: (my-lupine-project)
? 请选择 AI 平台: (opencode / claude) [opencode]

? 请指定代码仓库路径（可多个，用空格分隔）:
  ../my-app ../my-lib

✔  .lupine/ 已生成
✔  已关联 2 个仓库

下一步:
  cd .lupine
  opencode    # 或 claude
```

#### 参数接口

```bash
npx lupine init                         # 完全交互式
npx lupine init --name my-project       # 部分交互式
npx lupine init \
  --name my-project \
  --platform opencode \
  --repo ../my-app \
  --repo ../my-lib                       # 完全非交互式
npx lupine init --help                   # 帮助
```

#### 生成逻辑（`generate.js`）

1. 读取 `templates/` 目录下的模板文件列表
2. 对每个模板文件：
   - 替换占位符：`{project_name}` → 用户输入的项目名
   - 计算 checksum 写入 manifest
   - 输出到目标 `.lupine/` 对应路径
3. 生成 `.lupineconfig.json`（写入版本号、repo 路径等）
4. 生成 `.lupine-version`（记录当前版本号）

#### 占位符替换

当前支持的占位符：

| 占位符 | 替换值 | 出现位置 |
|--------|--------|---------|
| `{project_name}` | 用户输入的项目名 | CLAUDE.md, PRODUCT.md, README.md |
| `{project_name}` | 同上 |（同上） |

#### `init` 状态码

| 状态码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 目标目录 `.lupine/` 已存在（拒绝覆盖） |
| 2 | 交互中断（Ctrl+C） |
| 3 | 参数校验失败 |

#### 4.2.5 Agent 定义生成

`init` 命令还会根据用户选择的平台，生成对应的 Agent 定义文件。这是使 Lupine 多 Agent 架构能工作的关键。

**实现方式：**

- `templates/agents/*.prompt` — 4 个 Agent 的共用 prompt 正文（跨平台一致）
- `templates/agents/_agents.json` — 每个 Agent 的平台特定配置（opencode/claude 的 mode、permission、tools 等）
- `src/agents.js` — 读取 prompt + config，渲染为平台特定的 Agent 定义文件

**生成的文件（以 opencode 为例）：**

```
.opencode/agents/
├── lupine.md                mode: primary
├── lupine-planner.md        mode: subagent
├── lupine-executor.md       mode: subagent
└── lupine-evaluator.md      mode: subagent, temperature: 0.1
```

每个 Agent 定义文件格式：

```yaml
---
mode: subagent
permission:
  edit: deny
  bash:
    "*": ask
    "grep *": allow
description: Lupine 评估器，负责代码审查、设计审查、门禁检查
---
{prompt content}
```

**平台选择逻辑：** 用户在 `init` 时选择 opencode 或 claude，生成对应平台的 Agent 定义。当前默认为 opencode。

### 4.3 `update` 命令详细实现

#### 用户旅程

```bash
$ cd .lupine
$ lupine update

🐺 检查 .lupine/ 更新...

  当前版本: 1.0.0
  最新版本: 1.2.0

  ✔  CLAUDE.md                    (已更新)
  ✔  rules/agents.md              (已更新)
  ⏭  rules/coding.md              (已修改，跳过。使用 --force 覆盖)
  ⏭  rules/evals.md               (已修改，跳过)
  ✔  .lupine-version              (1.0.0 → 1.2.0)

  概要: 2 已更新, 2 已跳过, 0 失败
```

#### 参数接口

```bash
lupine update                          # 智能更新
lupine update --force                  # 全部覆盖（包括用户修改过的文件）
lupine update --force rules/coding.md  # 仅覆盖指定文件
lupine update --dry-run                # 预览，不实际写入
lupine update --help                   # 帮助
```

#### 保护策略（`checksum.js`）

1. `init` 时对每个模板文件计算 SHA256 哈希，写入 `_manifest.json`
2. `update` 时重新计算用户当前文件的 SHA256
3. 对比策略：

| 用户文件状态 | 行为 |
|-------------|------|
| checksum 匹配（未被修改） | ✅ 自动覆盖 |
| checksum 不匹配（被用户修改） | ⏭ 跳过，提示 `--force` |
| 目标文件不存在 | ✅ 新增 |
| 模板在新版中删除 | ⏭ 提示"已废弃"，不删除用户文件 |

#### `update` 状态码

| 状态码 | 含义 |
|--------|------|
| 0 | 全部成功 |
| 1 | 部分文件跳过（用户已修改） |
| 2 | 目标不是 `.lupine/` 或未初始化 |
| 3 | 已经是新版 |

#### 版本比较

采用语义化版本比较（semver）。`update` 仅在 `latest > current` 时执行。

### 4.4 模板自动同步（`sync-templates.js`）

#### 触发时机

- 发布 npm 前手动或 CI 自动执行
- 或框架规则文件变更后执行

#### 同步逻辑

```
项目根 rules/agents.md
  → 复制到 packages/lupine/templates/rules/agents.md
  → 替换占位符（{project_name} 等）

项目根 PRODUCT.md
  → 复制到 packages/lupine/templates/PRODUCT.md
  → 剔除 Lupine 自身产品定义，保留模板占位符结构

项目根 rules/constraints.yaml
  → 复制到 packages/lupine/templates/rules/constraints.yaml

同步后：
  → 对所有模板文件计算 SHA256
  → 写入 packages/lupine/templates/_manifest.json
```

#### 忽略列表

以下文件不应同步到用户模板：

| 文件 | 原因 |
|------|------|
| Lupine 自身 specs/ | 框架需求规格，非用户所需 |
| Lupine 自身 plans/ | 同上 |
| Lupine 自身 reviews/ | 同上 |
| Lupine 自身 CLAUDE.md | 框架自身 CLAUDE.md 是面向框架开发者的 |
| rules/constraints.yaml | ✅ 同步（这是用户需要自定义的） |

### 4.5 配置（`config.js`）

#### `.lupineconfig.json` 完整结构

```json
{
  "version": "1.0.0",
  "projectName": "my-project",
  "platform": "opencode",
  "repositories": [
    { "path": "../repo-a", "type": "code", "independentGit": true },
    { "path": "../repo-b", "type": "code", "independentGit": true }
  ]
}
```

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| version | string | ✅ | init 时写入的版本号 |
| projectName | string | ✅ | 用户输入的项目名称 |
| platform | string | ✅ | "opencode" 或 "claude" |
| repositories | array | ✅ | 关联的代码仓库列表 |
| repositories[].independentGit | boolean | | 是否独立 Git 仓库（自动检测目录下是否存在 .git） |

路径存储规则：
- 存储时保持用户输入的原始值（相对/绝对）
- 相对路径基准：`.lupine/` 所在目录的父目录（即用户执行 `lupine init` 时的目录）
- 不做路径规范化，保持透明

### 4.6 `.lupine-version` 文件

纯文本文件，仅包含版本号字符串：

```txt
1.2.0
```

`init` 时创建，`update` 时更新。用于版本检测，也作为"是否已初始化"的标记。

## 5. 发布流程

### 5.1 版本号管理

| 场景 | 命令 | 版本变化 |
|------|------|---------|
| 新功能 | `npm version minor` | 0.3.0 → 0.4.0 |
| Bug 修复 | `npm version patch` | 0.3.0 → 0.3.1 |
| 重大重构 | `npm version major` | 0.3.0 → 1.0.0（正式版） |

npm workspaces 自动同步根目录和 `packages/lupine/` 的版本号。

### 5.2 CI 发布流程

```yaml
# .github/workflows/release.yml
 on:
   push:
     tags:
       - 'v*'

 jobs:
   publish:
     runs-on: ubuntu-latest
     steps:
       - uses: actions/checkout@v4
       - uses: actions/setup-node@v4
         with:
           node-version: 20
           registry-url: 'https://registry.npmjs.org'

       - run: npm ci
       - run: npm run sync-templates    # 编译模板
       - run: npm publish -w packages/lupine  # 发 npm
         env:
           NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

       - run: ./scripts/build-release.sh  # 打 tarball
       - uses: softprops/action-gh-release@v1
         with:
           files: lupine-context-*.tar.gz
```

### 5.3 一次提交，双渠道发布

打 `v0.4.0` tag 推送到 GitHub，CI 自动完成：

```
git tag v0.4.0
git push --tags
  → CI:
    → npm publish lupine@0.4.0    (npm registry)
    → GitHub Release v0.4.0 + tarball    (GitHub Releases)
```

v0.x 阶段直接发 `latest` tag，不做 `@beta`。

## 7. 安全与约束

- **不操作仓库内容**：`lupine` 只记录仓库路径，不读取、不修改用户代码
- **不写 `~/.lupine/`**：无全局状态，所有数据在 `.lupine/` 本地目录
- **不自动执行代码**：无 postinstall 钩子
- **不收集数据**：无遥测、无网络请求（除了 npm install 本身和 update 的版本检查）
- **保护用户修改**：update 通过 checksum 保护用户已修改的文件

## 8. 边界情况

| 场景 | 行为 |
|------|------|
| `init` 时 `.lupine/` 已存在 | 退出码 1，提示"已初始化，使用 `lupine update` 更新" |
| `init` 时仓库路径不存在 | 警告但继续（路径由用户负责） |
| `update` 时 `.lupine/` 不存在 | 退出码 2，提示 `lupine init` |
| `update` 时版本相同 | 退出码 3，提示"已是最新版" |
| 用户中断交互（Ctrl+C） | 退出码 2，不写入任何文件 |
| 仓库路径是绝对路径 | 原样存储 |
| 仓库路径是相对路径 | 基于 `.lupine/` 父目录解析 |

## 10. 与现有框架的关系

### Python 版 `lupine-init` 的去留

| 阶段 | 状态 | 说明 |
|------|------|------|
| 当前 | 共存 | Node.js 版初版发布，Python 版保留不删 |
| 稳定后 | 标记为 deprecated | 在 README 和 help 中引导使用 Node.js 版 |
| 下个大版本 | 删除 | 移除 `framework/init/` 和 `bin/lupine-init` |

### tarball 分发的去留

tarball（GitHub Release）保留 — 它服务于不想用 npm 的用户（内网环境、不使用 Node.js 的团队）。但 tarball 的内容会从"Python 脚本 + 模板"变为"Node.js 包 + 模板"（用户同样需要 Node.js 来运行 `lupine`）。
