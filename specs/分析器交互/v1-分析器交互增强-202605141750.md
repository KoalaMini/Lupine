# 分析器交互增强

> 功能名称：分析器交互增强（Analyzer Brainstorming 增强）  
> 版本：v1.0  
> 创建时间：2026-05-14 17:50  
> 对应 VISION：Lupine 产品定位与愿景 v1.0

---

## 1. 功能概述

增强 Lupine 分析器（Analyzer）的需求澄清阶段，引入一问一答交互模式、HTML/SVG 草图能力、分块确认设计和多方案对比机制，使需求澄清过程更接近自然的协作式 brainstorming，降低用户认知负担，提升需求质量。

## 2. 目标用户

使用 Lupine Analyzer 进行需求分析的开发者/产品经理。

## 3. 功能范围

### 3.1 明确列入范围

1. **一问一答交互模式**
   - Analyzer 每次只问 1 个问题，不批量提问
   - 优先使用选择题（AskUserQuestion），降低用户回答成本
   - 需要开放回答时，用简单追问
   - 大项目（多独立子系统）先分解，不急于问细节

2. **HTML/SVG 草图能力**
   - Analyzer 在对话中生成 HTML/SVG 内容（架构图、流程图、UI 布局草图）
   - 支持多版本对比（如 `layout-A.svg` vs `layout-B.svg`）
   - 用户通过终端回复确认/选择
   - 不引入额外 HTTP server，Agent 直接生成文件

3. **文件路径规范**
   - 未确认草图：`.lupine/sketches/{功能名称}/`
   - 确认后草图：`specs/attachments/{功能名称}-v{版本号}-{YYYYMMDDHHmm}/`
   - 命名对齐对应 spec 文件

4. **分块确认设计**
   - 将设计按 section 展示：范围 → 架构 → 数据流 → 错误处理 → 测试策略
   - 每 section 用户确认后再继续下一 section
   - 任何 section 不通过则回退修改

5. **多方案对比**
   - 每个设计决策点提出 2-3 个方案
   - 用表格列出 trade-offs 和推荐方案
   - 草图支持并排展示对比

6. **YAGNI 自检**
   - 写 spec 前主动列出可疑功能，询问用户是否必要
   - 写 spec 后执行：placeholder 扫描、内部一致性检查、范围检查、歧义检查

### 3.2 明确不做

- 不引入浏览器/HTTP server（保持 CLI + Markdown 驱动）
- 不做实时协作编辑
- 不做 AI 自动决策（所有方案选择需用户确认）
- 不做复杂的前端渲染框架（HTML/SVG 为静态生成）

## 4. 详细设计

### 4.1 一问一答交互流程

```
用户提出需求
    |
    ▼
评估项目规模
    ├── 规模过大（多独立子系统）
    │       └── 分解为子项目 → 逐个 brainstorm
    └── 规模适中
            |
            ▼
    一问一答循环（每次 1 个问题）
            ├── 澄清问题（目的、约束、成功标准）
            ├── 2-3 方案对比（附 trade-offs）
            └── 草图辅助（如需要）
            |
            ▼
    分块确认设计
            ├── 范围 section → 确认？
            ├── 架构 section → 确认？
            ├── 数据流 section → 确认？
            ├── 错误处理 section → 确认？
            └── 测试策略 section → 确认？
            |
            ▼
    YAGNI 自检
            |
            ▼
    写入 spec 文件
            |
            ▼
    Spec 自检
            |
            ▼
    用户审查 spec → 确认？
            |
            ▼
    锁定版本，结束
```

### 4.2 草图生成规范

**SVG 生成要求：**
- 使用内联 SVG（`<svg>` 标签），可直接嵌入 markdown
- GitHub 原生支持 SVG 渲染
- 风格统一：线条简洁、配色克制（黑白灰为主，重点用单一强调色）

**草图类型：**
| 类型 | 用途 | 示例 |
|------|------|------|
| 架构图 | 展示系统组件关系 | 模块依赖、服务边界 |
| 流程图 | 展示用户/数据流程 | 注册流程、状态转换 |
| 布局草图 | 展示 UI 布局方案 | 页面结构、组件排列 |
| 对比图 | 并排展示方案差异 | A/B 布局对比 |

**文件管理：**
```
.lupine/sketches/分析器交互增强/           # 未确认
├── arch-v1.svg
├── arch-v1-B.svg                          # 方案 B 对比
└── flow-v1.svg

specs/attachments/分析器交互增强-v1.0-202605141750/  # 确认后
├── arch-final.svg
└── flow-final.svg
```

### 4.3 分块确认的数据结构

每个 section 包含：
- `title`: section 标题
- `content`: section 正文（markdown）
- `sketches`: [] 关联草图路径（可选）
- `status`: `pending` / `approved` / `rejected`
- `feedback`: 用户反馈（如拒绝，记录原因）

### 4.4 自检 Checklist

**YAGNI 自检（写 spec 前）：**
- [ ] 列出所有功能点
- [ ] 标记可疑/非必要功能
- [ ] 向用户逐一确认

**Spec 自检（写 spec 后）：**
- [ ] Placeholder 扫描：无 "TBD"、"TODO"、不完整 section
- [ ] 内部一致性：各 section 无矛盾
- [ ] 范围检查：聚焦单一功能，不过度发散
- [ ] 歧义检查：每个需求只能有一种理解方式

## 5. 安全与约束

- 草图文件不执行 JavaScript（纯 SVG，无 `<script>`）
- `.lupine/` 目录默认加入 `.gitignore`
- 用户确认后的附件随 spec 一起版本控制

## 6. 边界情况

| 场景 | 处理 |
|------|------|
| 用户不确认任何方案 | 回退到澄清阶段，重新提问 |
| 草图生成失败 | 降级为纯文本描述，不阻塞流程 |
| 用户中途退出 | 保存已确认 section，下次从断点恢复 |
| 需求中途大幅变更 | 视为新需求，重新开始 brainstorm |
| 单功能 spec 超过 3 次迭代 | 回溯到 VISION 层检查定位 |

## 7. 非功能要求

- 每轮一问一答响应时间 < 5 秒
- 草图生成时间 < 10 秒
- 支持中文内容生成
- 产出物符合 `rules/coding.md` 命名规范

## 8. 成功标准

- 使用新交互后，用户一次确认 spec 的比例 ≥ 60%（降低迭代次数）
- 草图辅助后，需求歧义导致的返工减少 ≥ 30%
- 一问一答模式下，用户放弃率低于批量提问模式
