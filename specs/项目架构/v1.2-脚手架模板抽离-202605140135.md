# 脚手架模板抽离与目录结构重组

## 版本

- v3（合并 v1.0 + v1.1）
- 日期: 2026-05-14
- 备注: 本版本包含对 `项目结构重组` 的修正（删除 `.lupine/`，框架迭代回归根目录），及后续追加的 Git Flow 策略和交互式确认

---

## 背景与问题

1. **模板硬编码在脚本中** — `bin/lupine-init` 用 heredoc 写死了所有初始化文件内容，修改模板需要改脚本，无法独立迭代
2. **`.lupine/` 职责混淆** — 用隐藏目录管理框架自身迭代，业界无此先例；`.claude/` 是工具输出目录，框架自身的业务文档不应走同样模式
3. **根目录的 `specs/`/`plans/` 本末倒置** — 当前放的是空 README 模板供 `lupine-init` 复制，而非 Lupine 框架自身的迭代文档
4. **框架没有吃自己的狗粮** — Lupine 倡导文档驱动开发，但自身的 VISION、SPECS、PLANS 却放在 `.lupine/` 中，与面向用户的成品解耦过度

## 目标

1. 模板文件从脚本中抽离为独立静态文件，支持独立迭代和覆盖式更新
2. Lupine 框架自身的需求/设计/审查/任务回归根目录，真正使用自身的四角色流程
3. 初始化脚本职责单一：从模板目录复制到目标项目
4. 被初始化项目与 Lupine 框架本身保持一致的目录结构

## 详细需求

### 1. 新建 `assets/` 目录存放模板

`assets/` 作为初始化模板仓库，目录结构与目标项目保持一致：

```
assets/
├── CLAUDE.md          # AI 索引入口模板（含占位符）
├── VISION.md          # 产品愿景模板（含占位符）
├── rules/
│   ├── agents.md      # Agent 角色定义与工作流
│   ├── coding.md      # 技术栈与编码规范（含占位符）
│   ├── evals.md       # 评估门禁 checklist
│   └── git.md         # Git 协作规范
├── specs/
│   └── README.md      # 规格说明书目录说明
├── plans/
│   └── README.md      # 设计方案目录说明
├── reviews/
│   └── README.md      # 审查报告目录说明
└── tasks/
    └── README.md      # 任务跟踪目录说明
```

- 模板内容从当前 `bin/lupine-init` 的 heredoc 中提取
- 文件命名与目标项目完全一致
- `assets/` 下的 README.md 使用与当前 `lupine-init` 生成的相同内容

### 2. 迁移 `.lupine/` 内容到根目录

将 `.lupine/` 下所有框架自身的迭代文档移动到根目录：

| 原路径 | 新路径 | 说明 |
|--------|--------|------|
| `.lupine/VISION.md` | `VISION.md` | 覆盖根目录当前的模板占位符 |
| `.lupine/specs/*` | `specs/` | 合并到根目录 specs |
| `.lupine/plans/*` | `plans/` | 合并到根目录 plans |
| `.lupine/reviews/*` | `reviews/` | 合并到根目录 reviews |
| `.lupine/tasks/*` | `tasks/` | 合并到根目录 tasks |

**冲突处理**：根目录当前 `specs/`、`plans/` 等下只有 README.md，与 `.lupine/` 下的 README.md 内容一致，直接覆盖即可。

### 3. 更新 `bin/lupine-init`

- 删除所有 heredoc 硬编码的模板内容
- 改为从 `assets/` 目录递归复制到目标项目
- 复制完成后，在目标目录创建 `README.md`（如果 assets 中没有，则生成一个基础版本）
- 保留当前的 skill 安装逻辑（检查并运行 `scripts/install-skills.sh`）
- 保留当前的用户提示和文件列表输出

```bash
# 核心逻辑示例
cp -r "$LUPINE_FRAMEWORK_DIR/assets/"* "$TARGET_DIR/"
```

### 4. 删除 `.lupine/` 目录

迁移完成后，删除 `.lupine/` 及其所有子目录和文件。

### 5. 更新项目文档

- **`CLAUDE.md`**：移除 `.lupine/` 相关说明，更新为"框架自身迭代直接使用根目录的 specs/plans/reviews/tasks"
- **`README.md`**：更新文件结构说明，加入 `assets/` 目录的说明
- **`rules/` 中的文件**：如有引用 `.lupine/` 或 `PLANS/` 等大写路径，统一更新

#### 需求 6：更新 Git 分支策略

**框架自身 `rules/git.md`** 和 **模板 `assets/rules/git.md`** 同时更新为：

```
master:   保护分支，仅接受 release 的 PR，打 tag 发布
dev:      日常开发集成分支
release:  从 dev 拉出，测试后 PR → master
feat/{功能名}:  从 dev 拉出，开发完成后 PR → dev（功能名优先使用中文）
fix/{修复名}:   从 dev 拉出，修复完成后 PR → dev（修复名优先使用中文）
```

Commit 规范、PR 流程保持不变。

#### 需求 7：lupine-init 交互式确认

在 `bin/lupine-init` 的复制流程中新增一步：

```
Step 4.5（在复制模板之后，在安装 skills 之前）:
  询问用户: "是否采用 Lupine 推荐的 Git Flow 分支策略？(Y/n)"
  - 用户接受 (Y/回车) → 保留 assets/rules/git.md（含 Git Flow 内容）
  - 用户拒绝 (n/N) → 将已复制的 rules/git.md 清空为仅含标题的空文件
```

**空的 `rules/git.md` 内容示例：**

```markdown
# Git 协作规范

（此项目未采用 Lupine 推荐的 Git Flow，请根据团队规范自行补充）
```

## 边界情况

| 场景 | 处理方式 |
|------|---------|
| `lupine-init` 目标目录已存在同名文件 | 覆盖（与现有行为一致，用户已确认覆盖式更新策略） |
| `assets/` 目录缺失 | 脚本报错退出，提示"模板目录不存在，请检查 Lupine 框架安装" |
| 框架自身继续迭代 | 直接在根目录 `specs/` 下创建新的规格说明书，与任何其他使用 Lupine 的项目完全一致 |
| 用户输入大小写不敏感（n/N/no/NO） | 统一用 `[Nn]*` 模式匹配 |
| 用户输入任意其他字符（含回车） | 默认为 "Y"，保留 Git Flow |
| `lupine-init` 非交互环境（CI 管道） | 无 stdin 时 `read` 读取空字符串 → 落入默认分支 → 保留 Git Flow |
| 用户拒绝后自行补充 git.md | 空文件保留格式骨架，用户后续编辑即可 |
