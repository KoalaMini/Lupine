# 产出物与角色模型

> Lupine 新一代产品架构  
> 版本：v1.0  
> 创建时间：2026-05-26

---

## 1. 概述

### 1.1 背景

原架构存在三个核心问题：

1. **产出物冗余**：tasks/ 目录中的任务跟踪表无人阅读，执行器的真实产出是代码而非文档
2. **边界模糊**：reqs（需求规格）和 plans（技术方案）内容重叠，读者不知道以哪个为准
3. **流程僵化**：四子 Agent 线性流水线（探索 → 规划 → 执行 → 评估）不匹配实际开发中灵活跳转的需求

### 1.2 解决思路

- **精简产出物**：取消 tasks/，明确 specs 和 plans 为**并行文档类型**，而非上下游关系
- **Agent 能力模型**：将 MCP、Skill、Constraints 定义为 Agent 的可插拔能力层，推荐制 + 用户自定义
- **灵活调度**：Lupine 作为调度器自主决策，仅约束角色边界，不预设路径

### 1.3 范围

| 列入 | 不做 |
|------|------|
| 产出物模型（PRODUCT.md / specs / plans）定义 | 不定义具体约束内容（由用户自定义） |
| 角色边界与能力模型（Prompt + MCP/Skill + Constraints） | 不涉及具体 MCP/Skill 的实现细节 |
| Lupine 调度决策框架 | 不预设路径模板 |
| 约束机制定义 | 不涉及界面/CLI 工具开发 |

---

## 2. 产出物模型

产出物分为三类文档，各司其职：

### 2.1 PRODUCT.md（产品总纲）

产品级顶层文档，长期不变。记录产品定位、核心理念、目标用户、成功标准、范围边界。

- **位置**：项目根目录
- **所有者**：Lupine
- **更新频率**：仅当产品定位/范围发生变化时

### 2.2 specs/（需求规格）

给人读的需求文档，回答 WHAT 和 WHY。面向复杂场景（产品需求、模糊需求澄清）。

- **位置**：`specs/{分类}/{版本号}-{功能名称}-{YYYYMMDDHHmm}.md`
- **所有者**：仅 Lupine 可写
- **读者**：人（PM、开发者审查用）
- **版本规则**：
  - 首次定稿为 `v1.0`
  - 小修改版本号末位 +1（如 `v1.0` → `v1.1` → `v1.2`）
  - 大幅重构复位为 `v2.0`
  - 文档版本号与产品版本号独立

**规格文件结构：**

```markdown
---
spec: <分类>
version: <版本号>
title: <标题>
status: draft | confirmed | superseded
date: <创建日期>
supersedes: <被取代的旧 spec>
changes:
  - <变更记录>
---

# <标题>

> 版本：v<版本号>
> 创建时间：<创建时间>

---

## 1. 功能概述

## 2. 目标用户

## 3. 功能范围（列入 / 不做）

## 4. 详细设计

## 5. 安全与约束

## 6. 边界情况

## 7. 非功能要求

## 8. 成功标准
```

### 2.3 plans/（执行计划）

给 AI 执行的指令文档，回答 HOW。面向所有执行动作：新功能实现、交互优化、Bug 修复。

- **位置**：`plans/{YYYYMMDDHHmm}-{功能名称}.md`
- **所有者**：仅 Planner 可写
- **读者**：Executor（主要）+ Evaluator（审查基线）
- **核心属性**：**只读**。Executor 不得修改 plan 内容（步骤列表、文件清单、架构描述等结构性内容不可变）。
  - 唯一的例外：**checkbox 状态更新**（`- [ ]` → `- [x]`）。这是进度追踪，不被视为内容修改。

**计划文件结构：**

```markdown
---
plan: <功能名称>
readonly: true
ref_spec: specs/<分类>/v<版本>-<名称>-<时间戳>.md  # 可选
steps: <总步数>
created: <创建时间>
---

# <功能名称> 执行计划

> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax for tracking.  
> 规划时间：<创建时间>

---

## 目标

## 技术方案（简述）

## 文件清单

| 文件 | 操作 | 说明 |
|------|------|------|

## 执行步骤

- [ ] **Step 1: <步骤名>**

  目标：<此步骤要达成的结果>

  [[具体指令]]

### 引用关系

plan 可引用 spec（`ref_spec: specs/分类/v1.0-xxx.md`）。如果实现过程中发现 spec 需要更新，Planner **同步更新 spec 版本**（Lupine 确认后写入新的 v1.1）。

### 并行文档模型示意

```
PRODUCT.md            产品总纲（长期不变）
  │
  ├── specs/          需求规格（给人读，WHAT + WHY）
  │   ├── 用户系统/v1.0-用户登录-202605201430.md
  │   ├── 用户系统/v1.1-用户登录-202605221000.md
  │   └── 支付模块/v1.0-支付流程-202605251600.md
  │
  └── plans/          执行计划（给AI执行，只读，HOW）
      ├── 202605201430-实现用户登录.md      (ref: specs/用户系统/v1.0)
      ├── 202605221000-优化登录交互.md       (ref: specs/用户系统/v1.1)
      └── 202605261200-修复支付回调Bug.md
```

spec 和 plan 不是上下游关系，而是两种独立的文档产物：
- Lupine 可以只产 spec 不产 plan（需求确定但未排期）
- Planner 可以只产 plan 不涉及 spec（纯优化 / Bug 修复）
- plan 可以引用 spec 并触发 spec 版本更新

## 3. 角色与能力模型

### 3.1 角色定义

| 角色 | 类型 | 职责 | 产出物 |
|------|------|------|--------|
| Lupine | 主 Agent | 产品经理，跟用户对话探讨、写 specs、调度子 Agent | specs/ |
| 探索器 Explorer | 子 Agent | 代码库探索、Web 搜索、文档阅读（只读） | 结构化情报摘要 |
| 规划器 Planner | 子 Agent | 基于需求（spec 或口头需求）写执行计划 | plans/ |
| 执行器 Executor | 子 Agent | 按 plan 写代码、写测试 | code + tests |
| 评估器 Evaluator | 子 Agent | 审 plan + 审 code，门禁控制 | reviews/ |

### 3.2 Agent 三层结构

每个 Agent 的定义由三层组成：

```
┌─────────────────────────────────────┐
│          Agent Prompt               │
│  角色定义、职责描述、工作流规则       │
├─────────────────────────────────────┤
│          MCP 依赖清单（推荐）         │
│  如：filesystem, browser-use        │
├─────────────────────────────────────┤
│          Skill 依赖清单（推荐）       │
│  如：lupine-diagram, writing-plans  │
├─────────────────────────────────────┤
│       Constraints（推荐 + 自定义）    │
│  如：冒烟测试、回归测试              │
└─────────────────────────────────────┘
```

### 3.3 MCP/Skill 推荐清单

推荐制（recommended）含义：
- 系统为每个角色预设推荐清单
- 自动检测用户已安装的 MCP/Skill 并合并
- 用户可以覆盖或自定义

```yaml
# 推荐 MCP
mcp_recommended:
  evaluator:
    - playwright-mcp    # 强烈推荐：用于冒烟/回归/功能/边界值等测试
  executor:
    - filesystem        # 文件操作
  explorer:
    - websearch         # 搜索调研
    - playwright-mcp    # 访问网页

# 推荐 Skill
skill_recommended:
  planner:
    - lupine-diagram    # 架构图/流程图
    - writing-plans     # 写可执行计划
  executor:
    - writing-plans     # 拆 task、管理 checkbox
  evaluator:
    # （约束承担了 skill 的职责）
```

> 注意：上述清单为初始推荐，随社区 MCP/Skill 生态演进持续更新。

### 3.4 约束集成

约束见第 5 节详细定义。

## 4. 流程调度模型

### 4.1 角色边界（硬规则）

以下边界**在任何场景下都不可打破**：

| 规则 | 说明 |
|------|------|
| **产出物所有权** | 仅 Lupine 写 specs，仅 Planner 写 plans，仅 Executor 写代码，仅 Evaluator 写 reviews |
| **plan 只读** | Executor 可以更新 checkbox 状态，但不能修改 plan 内容 |
| **plan 缺陷回退** | Executor 发现 plan 不可行 → 停等，回 Planner 修改 → Evaluator 再审 |
| **spec 歧义回退** | 评估器发现 spec 歧义或遗漏 → 回 Lupine 澄清 |
| **禁止跳评估** | 不允许跳过 Evaluator 审查直接合并代码 |
| **禁止自审** | Executor 不能审自己的代码，Planner 不能审自己的 plan |

### 4.2 Lupine 决策框架

当用户输入请求时，Lupine 按以下三步决策：

```
步骤 1：分类
  ├── 场景1（产品需求）—— 用户有明确的功能规划
  ├── 场景2（模糊需求）—— 用户只有想法，不清晰
  ├── 场景3（交互优化）—— 改动点明确、范围小
  └── 场景4（修复Bug）—— 线上问题或缺陷

步骤 2：判定产出物需求
  ├── 需要 spec？→ Lupine 跟用户探讨后写
  ├── 需要 plan？→ 派 Planner
  ├── 需要调研？→ 先派 Explorer 再决策
  └── 两个都要？→ 先写 spec，Planner 后续独立写 plan

步骤 3：调度子 Agent
  └── Lupine 自主决定顺序、是否并行、是否跳过
```

> 此决策框架为**思考框架**（thinking framework），非硬编码决策树。Lupine 可以基于上下文灵活调整。

### 4.3 常见场景的典型调度（非预设路径）

| 场景 | Lupine 的典型决策 | 产出物 |
|------|------------------|--------|
| 场景1 产品需求 | 对话探讨 → 写 spec → 派 Planner → 审 plan → 派 Executor → 审 code | spec + plan |
| 场景2 模糊需求 | 派 Explorer 调研 → 对话澄清 → 写 spec → 派 Planner → ... | spec + plan |
| 场景3 交互优化 | 派 Planner 写 plan → 审 plan → 派 Executor → 审 code | plan（可选更新 spec） |
| 场景4 修 Bug | 复现确认 → 派 Planner 写 plan → 审 plan → 派 Executor → 审 code | plan（可选补 spec） |
| 微小改动 | 直接派 Executor → 审 code | 仅代码 |
| 紧急修复 | 直接派 Executor → 审 code → 事后补 plan + spec | 事后补文档 |

## 5. 约束机制

### 5.1 定义

约束（Constraint）是一段注入 Agent prompt 的指导原则，告诉 Agent 在履行职责时应遵循哪些质量要求。

约束**不是独立的可执行程序**，它以自然语言描述注入到 Agent 的系统提示中。

### 5.2 存储

约束定义在 `rules/constraints.yaml` 中：

```yaml
# rules/constraints.yaml
# 推荐约束定义
# 用户可按需修改、增删

constraints:
  evaluator:
    - name: 冒烟测试
      desc: 核心功能是否跑得通
    - name: 回归测试
      desc: 改代码后旧功能没坏
    - name: 功能测试
      desc: 功能是否按需求规格工作
    - name: 边界值测试
      desc: 输入在边界上的表现
    - name: 等价类划分
      desc: 用代表性输入覆盖所有类别

  executor:
    - name: 遵循计划
      desc: 严格按照 plan 的步骤执行，不擅自偏离
    - name: 步骤追踪
      desc: 每完成一步更新 checkbox 状态
    - name: 计划缺陷上报
      desc: 遇到 plan 不可行时停等，回 Planner 修改

  planner:
    - name: 可执行粒度
      desc: 每个 step 必须具体到单一可独立执行的任务单元
    - name: 验收条件
      desc: 每个 step 应包含明确的完成标准
    - name: 只读约束
      desc: plan 确认后标记为 readonly: true

  explorer:
    - name: 只读
      desc: 不改任何文件
    - name: 结构化返回
      desc: 结果以结构化摘要返回，标注置信度
```

### 5.3 注入机制

约束在 Agent 被调用时，自动拼接到其 prompt 中：

```
你是评估器。
你的职责是审查 plan 和 code。

## 约束
请遵循以下审查标准：
1. 冒烟测试 — 核心功能是否跑得通
2. 回归测试 — 改代码后旧功能没坏
3. 功能测试 — 功能是否按需求规格工作
4. 边界值测试 — 输入在边界上的表现
5. 等价类划分 — 用代表性输入覆盖所有类别
```

### 5.4 三阶叠加

| 优先级 | 来源 | 行为 |
|--------|------|------|
| 低 | 系统推荐（配置文件中的默认值） | 作为 baseline 注入 |
| 中 | 用户已安装/已配置的约束 | 自动合并 |
| **高** | 用户对约束文件的显式修改 | 以用户修改为准 |

**变动对比（相对原架构）：**

| 变动 | 原 | 新 |
|------|-----|-----|

| 约束管理 | 无独立文件 | → rules/constraints.yaml |
| plans | 线性依赖 spec | 与 spec 并行，独立演进 |

## 7. 成功标准

| 标准 | 衡量方式 |
|------|---------|
| specs 和 plans 不再内容重叠 | 随机抽取 3 组，检查有无对方域的内容 |
| 评估器审 code 有明确的 plan 基线 | plan 文件引用率 ≥ 90% |
| 约束注入后评估质量可感知 | 用户可说出"评估器按了哪几条约束在审" |
| 调度灵活度 | 同一场景下不同次，Lupine 可根据上下文走不同路径（不机械重复） |
